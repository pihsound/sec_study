1
1991년, 한 프로그래머의 분노
Phil Zimmermann

1991년 미국. 인터넷은 빠르게 퍼져나가고 있었고, 이메일은 이미 일상이 됐어요. 그런데 이메일의 본질은 여전히 엽서였어요 — 전달 경로의 누구나 읽을 수 있는.

필 짐머만(Phil Zimmermann)은 이걸 보다 못했어요. 특히 그는 정치 활동가·반핵 운동가들을 많이 알고 있었는데, 이들이 이메일로 주고받는 민감한 내용이 감시당할 수 있다는 게 견딜 수 없었어요.

"인터넷에서 프라이버시는 오픈 사회에서 필수적인 것이다. 나는 무언가를 만들어야 했다."
— Phil Zimmermann, PGP 개발 동기

그래서 그는 PGP(Pretty Good Privacy)를 만들어 인터넷에 무료로 공개했어요. 이름도 소박해요 — "꽤 괜찮은 프라이버시". 과대 선전 없이, 그냥 필요해서 만들었다는 태도가 느껴져요.

재밌는 건, 이 공개가 미국 정부를 발끈하게 만들었다는 거예요. 당시 미국은 강력한 암호화 기술을 무기로 분류해 수출을 통제하고 있었어요. 짐머만은 3년간 FBI 수사를 받았고, 결국 기소되지는 않았지만 PGP는 이미 전 세계에 퍼진 후였죠.

2
PGP의 핵심 문제 의식 — 두 가지 질문
기술적 배경

PGP를 만들면서 짐머만은 두 가지 근본적인 질문을 해결해야 했어요.

❓ 질문 1 — "어떻게 빠르게 암호화하나?"

공개키 암호(RSA)는 느려요. 1MB짜리 이메일을 RSA로 암호화하면 시간이 너무 오래 걸려요. 그렇다고 빠른 대칭키를 쓰자니, 키를 어떻게 안전하게 전달할까요? 상대방과 만나서 직접 교환해야 하나요?

해결: 하이브리드 암호화 — 메시지 자체는 빠른 대칭키로 암호화하고, 그 대칭키(세션키)만 상대방의 공개키로 잠가서 함께 보낸다.

❓ 질문 2 — "공개키가 진짜인지 어떻게 믿나?"

누군가가 "이게 김철수의 공개키야"라고 보내줬어요. 근데 이게 진짜 김철수의 키인지 어떻게 알아요? 공격자가 중간에 끼어 가짜 키를 심어뒀을 수도 있잖아요 (중간자 공격, MITM).

기존 해결책: CA(인증기관)가 "이 키는 진짜야"라고 보증. 근데 CA를 믿으려면 CA의 공개키를 믿어야 하고... 결국 누군가를 맹목적으로 믿어야 해요.

PGP의 해결: Web of Trust — "내가 직접 아는 사람의 공개키를 믿는다. 그 사람이 믿는 사람도 믿는다." 신뢰를 사람들 사이에 분산시켜요.

3
Web of Trust — 신뢰의 거미줄
PGP의 철학

Web of Trust를 이해하려면 현실 사회의 신뢰 구조를 떠올리면 돼요.

나는 친구 철수의 공개키를 직접 만나서 확인했어요. 그래서 철수의 키를 믿어요. 철수는 영희의 키를 확인하고 "이건 진짜 영희 키야"라고 서명을 해줬어요. 나는 철수를 믿으니까, 철수가 서명한 영희의 키도 믿을 수 있어요. 영희는 또 박사님의 키를 서명했고...

Web of Trust — 분산 신뢰의 거미줄

CA 방식(S/MIME, HTTPS)과의 결정적 차이는 이거예요.

PGP — Web of Trust (분산)

신뢰를 사람들 사이에 분산. CA 같은 중앙 권위 없음. 내가 직접 아는 사람들이 서로 키를 서명해줘서 신뢰가 퍼져나감.

✅ 중앙 서버 불필요 / ✅ 무료 / ✅ 검열 어려움
❌ 모르는 사람과는 바로 암호화 통신 불가 / ❌ 공개키 서버 운영 필요

CA 방식 — 중앙 집중 신뢰

DigiCert, Let's Encrypt 같은 인증기관(CA)이 "이 키는 진짜야"라고 보증. 모든 신뢰가 CA로 수렴. 처음 보는 사이트도 바로 신뢰 가능.

✅ 편리함 / ✅ 확장성 / ✅ 처음 만나는 상대도 OK
❌ CA 해킹 시 전체 붕괴 / ❌ CA를 맹목적으로 신뢰해야 함

4
PEM — 기관들의 표준화 시도
1993년, IETF

PGP가 개인들 사이에서 폭발적으로 퍼지던 시절, IETF(인터넷 표준화 기구)에서도 이메일 보안 표준을 만들려고 했어요. 그게 PEM(Privacy Enhanced Mail)이에요.

PEM의 접근법은 PGP와 정반대였어요.

PGP: "CA 같은 중앙 권위 없이 사람들이 서로 믿으면 돼"
PEM: "표준화된 CA 인프라를 통해 체계적으로 신뢰를 관리하자"

PEM은 기술적으로는 PGP보다 더 완성도 높고 보안 능력도 뛰어났어요. 이진 데이터를 Base64로 변환해 전송하는 방식도 체계적이었고요.

근데 문제가 있었어요. 중앙 집중식 CA 인프라를 실제로 구축하기가 너무 어려웠어요. 전 세계 인터넷 사용자가 전부 가입해야 하는 CA 시스템을 1990년대 초반에 만드는 건 현실적으로 불가능에 가까웠죠.

결국 PEM은 이론으로는 훌륭했지만 실제 채택은 거의 안 됐고, PGP와 그 뒤를 이은 S/MIME이 실용적인 이메일 보안을 담당하게 됐어요.

💡 역사가 주는 교훈

PGP와 PEM의 역사는 기술 표준화의 딜레마를 보여줘요. 기술적으로 완벽한 것이 이기는 게 아니라, 현실적으로 사람들이 쓸 수 있는 것이 이겨요. PGP는 불완전하지만 개인이 바로 쓸 수 있었고, PEM은 훌륭하지만 인프라가 없으면 쓸 수 없었어요.

PEM 정의 및 배경
Definition
PEM 공식 정의

PEM(Privacy Enhanced Mail)은 인터넷 표준안으로 IETF에서 만든 이메일 암호화 기법. 이진 데이터를 6비트씩 분할하여 아스키 문자로 변환하는 Base64 인코딩 방식 사용. RFC 1421~1424에 정의. 중앙 집중식 키 인증 방식으로 CA 인프라 필요.

대중화

암호화

2
PEM 특징 — 왜 실패했나
Analysis
✓
강점 — 기술적으로는 PGP보다 완성도 높음
자동 암호화로 전송 중 유출돼도 내용 확인 불가. 체계적인 CA 인프라 기반 신뢰. Base64로 표준화된 인코딩. 이론적 보안성 우수.
✗
약점 — 인프라 구축 불가능이 발목을 잡음
전 세계 인터넷 사용자가 가입해야 하는 중앙 집중식 CA 인프라를 1990년대 초반에 구축하는 건 현실적으로 불가능. 결국 채택률 극히 낮음. 대중화 실패.
3
Base64 인코딩 원리
이해
문제: SMTP는 7비트 ASCII만 처리 가능. 이진 암호문은 8비트 데이터.
해결: Base64 — 이진 데이터를 ASCII 문자로 변환

변환 방법:
원본 이진 데이터: 01001000 01100101 01101100
6비트씩 분할: 010010 000110 010101 101100
↓ ↓ ↓ ↓
Base64 문자: S G V s

결과: 이진 3바이트(24비트) → ASCII 문자 4개 (33% 증가)
문자셋: A-Z(26) + a-z(26) + 0-9(10) + + + / = 64가지
항목Base64 특성
분할 단위6비트씩 분할
사용 문자A-Z(26) + a-z(26) + 0-9(10) + "+" + "/" = 64가지
패딩원본이 3바이트 배수 아닐 때 "=" 패딩 추가
크기 변화원본 대비 약 33% 증가
목적이진 데이터 → ASCII → SMTP 전송 가능
PGP와 비교PGP는 Radix-64 = 사실상 Base64와 동일 (CRC 추가)

S/MIME 탄생 배경 — PGP의 단점을 기업이 해결하다
스토리

PGP가 세상에 나왔을 때 개인 사용자들은 열광했어요. 하지만 기업 IT 담당자들의 반응은 달랐어요.

"Web of Trust? 처음 이메일 보내는 거래처 직원이랑 어떻게 미리 공개키를 교환해요?
우리 직원이 3,000명인데 전부 PGP 키 관리를 직접 하라고요?"
→ 기업 환경에서 PGP의 분산 신뢰 모델은 너무 비실용적이었어요.

1995년, RSA Data Security가 이 문제를 해결하기 위해 MIME 포맷 위에 보안을 얹은 S/MIME(Secure/Multipurpose Internet Mail Extensions)을 발표했어요. 핵심 아이디어는 단순해요.

"모든 신뢰를 CA(인증기관)가 보증하자. 처음 만나는 상대방이라도 CA가 서명한 인증서만 있으면 바로 암호화 통신이 가능하다."

이것이 이미 웹(HTTPS)에서 검증된 X.509 인증서 체계를 이메일에 그대로 가져온 거예요. 덕분에 기업 이메일 서버에 S/MIME을 구축하면, 직원들이 키 관리를 신경 쓸 필요 없이 이메일 암호화를 사용할 수 있어요.

💡 S/MIME이 기업 표준이 된 이유

Outlook, Gmail(Google Workspace), Apple Mail 등 주요 이메일 클라이언트가 모두 S/MIME을 기본 지원해요. CA에서 인증서를 발급받으면 추가 플러그인 없이 바로 사용 가능. IT 관리자 입장에서는 직원 전체에게 인증서를 발급하고 중앙 관리할 수 있는 S/MIME이 훨씬 편했어요.

S/MIME 공식 정의

S/MIME(Secure/Multipurpose Internet Mail Extensions)은 RSA Data Security가 MIME 인터넷 메일 형식을 보안 강화시킨 전자우편 보안 시스템. IETF 작업 그룹에서 표준화(RFC 3850·3851). X.509 버전 3 공개키 인증서를 사용하는 중앙 집중식 신뢰 모델. 기밀성·무결성·사용자 인증·송신 부인방지 제공.

S/MIME 데이터 유형 4가지
개념
①
봉함된 데이터 (Enveloped Data)
임의 유형의 암호화된 내용(기밀성) + 하나 또는 여러 수신자를 위해 암호화된 내용 암호키로 구성. 메시지를 대칭키(Triple-DES)로 암호화하고, 그 대칭키를 수신자의 공개키로 암호화하여 함께 전송. S/MIME을 갖춘 수신자만 열람 가능.
기밀성
②
서명된 데이터 (Signed Data)
서명될 내용의 메시지 다이제스트를 구하고 서명자의 개인키로 암호화하여 전자서명 생성. 서명된 데이터 메시지는 S/MIME 기능을 갖춘 수신자만 서명을 검증할 수 있고 내용을 볼 수 있음. Base64로 인코딩됨.
무결성·인증·부인방지
③
명문-서명 데이터 (Clear-Signed Data)
서명된 데이터로서 내용에 대한 서명만 생성(내용 자체는 암호화 안 함). 전자서명만 Base64로 인코딩. S/MIME 기능이 없는 수신자도 내용을 읽을 수는 있지만 서명을 검증할 수는 없음.
서명만 (내용은 평문)
④
서명되고 봉함된 데이터 (Signed and Enveloped Data)
서명된 데이터와 봉함된 데이터를 중첩한 형태. 암호화된 데이터는 서명될 수 있고, 서명된 데이터는 암호화될 수도 있음. 기밀성 + 무결성 + 인증 + 부인방지를 모두 제공.
기밀성·무결성·인증·부인방지 모두

S/MIME vs PGP — 실질적 차이
★★★
S/MIME vs PGP — 신뢰 체계 비교
⚠S/MIME 핵심 함정 — 어떤 알고리즘이 무엇에 쓰이나
"S/MIME의 메시지 무결성은 Triple-DES로 제공한다" → ✗ 오답

Triple-DES = 메시지 암호화(기밀성). SHA-1 = 무결성(해시함수). DSA = 부인방지(전자서명). X.509 v3 = 사용자 인증. 각 서비스별 알고리즘 매핑을 정확히 외워야 해요.

"S/MIME은 인증기관(CA) 없이 Web of Trust로 동작한다" → ✗ 오답

Web of Trust = PGP. S/MIME = X.509 v3 인증서 + CA 필수.

"S/MIME의 명문-서명 데이터는 S/MIME 없이도 내용을 읽을 수 없다" → ✗ 오답

명문-서명(Clear-Signed) = 내용은 암호화 안 됨 → S/MIME 없어도 내용은 읽을 수 있음. 단, 서명 검증은 S/MIME 필요.

📄 2014 국가직 9급 · PGP 하이브리드 암호화★★★
PGP(Pretty Good Privacy)에 대한 설명으로 옳지 않은 것은?
① PGP는 전자우편 보안 프로토콜이다.
② 암호화하지 않고, 암호화된 방법으로 메시지를 전송한다.
③ PGP는 압축을 한다.
④ Phil Zimmermann이 개발하였다.

① 이메일 보안이나 파일 암호화에 사용된다.
② 공개키 인증을 위해 PGP 인증서를 사용한다.
③ 자신의 공개키를 전달하는 데 인증기관의 서명이 필요하다.
④ 이메일에 서명할 때, 서명자의 패스워드를 요구한다.
⑤ 이메일 관리 프로그램에 플러그인도 가능하다.
정답 ③ — PGP의 핵심 특징: CA(인증기관) 없이 Web of Trust. "인증기관 서명 필요" = S/MIME의 특성. PGP는 사용자 개인이 공개키를 직접 교환하거나 공개키 서버에 올림.

① E-mail의 보안과 파일 암호화에 사용된다.
② 전자 서명, 기밀성, 압축, 단편화와 재조립 등의 기능이 있다.
③ E-mail 응용 프로그램에 플러그인도 가능하다.
④ 자신의 공개키를 전달하려면 인증기관의 서명이 필요하다.

① PGP
② PEM
③ S/MIME

📄 2020 경찰공무원 · PGP X.509 사용 불가 오답★★

② 전자우편에서 프라이버시, 무결성, 인증을 제공한다.
③ 인증기관의 인증서가 필요하지 않다.
④ 계층적인 신뢰 구조에 따라 X.509 인증서는 사용할 수 없다.
정답 ④ — PGP의 기본 신뢰 모델은 Web of Trust이지만, X.509 인증서를 아예 사용할 수 없는 것은 아니에요. PGP도 X.509를 지원할 수 있어요. "사용할 수 없다"는 표현이 오답.
📄 2020 경찰간부 · PEM 분산 방식 오답★★★
전자우편 보안에 사용되는 기술들에 대한 설명으로 가장 옳지 않은 것은?
① PGP는 전자우편 메시지의 인증과 기밀성 제공을 위해 개발된 전자우편 보안의 표준안이다.
② S/MIME이 제공하는 보안 서비스에는 기밀성, 무결성, 송신 사실 부인방지, 사용자 인증 등이 있다.
③ S/MIME은 X.509 버전3 규정을 준수하여 공개키 인증서를 사용한다.
④ PEM(Privacy Enhanced Mail)은 분산화된 키 인증 방식을 통해 관리가 용이하여 대중적으로 사용되고 있다.
정답 ④ — PEM은 중앙 집중식 키 인증 방식. "분산화" = PGP(Web of Trust)의 특성. PEM은 중앙 집중식이라 구축 어려워 대중적 사용이 어렵다. ④는 두 가지가 반대로 서술됨.

다음을 설명하는 전자우편 암호화와 가장 가까운 것은?

1991년에 IDEA 알고리즘과 RSA 알고리즘을 조합하여 만들어졌다. 세션 키를 암호화하기 위해 IDEA 알고리즘을 이용하고 사용자 인증을 위한 전자 서명에는 RSA 알고리즘을 이용한다. 특히 해당 기술을 사용하는 사람들 간의 신뢰 관계를 통해 공개키를 인증하는 기법을 사용하고 있다.
① S/MIME
② PGP
③ PEM
④ SSL
정답 ② — 키워드 3개: "1991년" + "IDEA+RSA" + "사람들 간의 신뢰 관계(Web of Trust)" = PGP. SSL은 이메일 보안이 아님. PEM은 IETF 표준이고 중앙 집중식.
📄 복합 함정 — PGP 처리 순서 (서명→압축→암호화)★★★
PGP의 전자서명·압축·암호화 처리 순서로 올바른 것은?
① 압축 → 전자서명 → 암호화
② 암호화 → 전자서명 → 압축
③ 전자서명 → 압축 → 암호화
④ 전자서명 → 암호화 → 압축
정답 ③ — 서명 먼저: 원본에 서명해야 나중에 검증 가능. 압축 그 다음: 암호화 전에 압축해야 크기 감소 효과. 암호화 마지막: 압축된 데이터를 세션키로 암호화. 암기: 서 → 압 → 암.
📄 2018 국회직 9급 · S/MIME 설명 매칭★★★
IETF의 작업 그룹에서 RSADSI의 기술을 기반으로 개발한 전자우편 보안 기술이며 RFC 3850, 3851 등에서 정의. 전자우편에 대한 암호화 및 전자서명을 통하여 메시지 기밀성, 메시지 무결성, 사용자 인증, 송신 사실 부인 방지, 프라이버시 보호 등의 보안 기능을 제공한다.
① MIME
② SMTP
③ PGP
④ PEM
⑤ S/MIME
정답 ⑤ — 키워드: "IETF + RSADSI 기술 + RFC 3850·3851 + 암호화·서명·기밀성·무결성·인증·부인방지" = S/MIME. PGP는 Phil Zimmermann 개발·1991년·CA 없음. PEM은 IETF이지만 RFC 1421~1424이고 중앙집중식. MIME은 보안 없는 이메일 확장.
📄 2021 군무원 7급 · S/MIME 보안 서비스 알고리즘 매핑★★★
S/MIME이 제공하는 보안 서비스와 알고리즘이 옳지 않게 연결된 것은?
① 메시지 암호화 — Triple-DES
② 메시지 무결성 — Triple-DES
③ 사용자 인증 — X.509 v3 인증서
④ 송신 부인방지 — DSA
정답 ② — 메시지 무결성 = SHA-1 (해시함수). Triple-DES는 메시지 암호화(기밀성)에 사용. 알고리즘 매핑 정확히 암기: 기밀성=3DES / 무결성=SHA-1 / 인증=X.509v3 / 부인방지=DSA.
⚠함정 정리 — PGP·PEM·S/MIME 출제 패턴
"PGP는 인증기관(CA)의 서명이 필요하다" → ✗ PGP = CA 없음, Web of Trust
"PEM은 분산화된 키 인증 방식" → ✗ PEM = 중앙 집중식 (분산은 PGP)
"PGP 처리 순서: 압축 → 서명 → 암호화" → ✗ 서명 → 압축 → 암호화
"S/MIME 무결성 = Triple-DES" → ✗ 무결성=SHA-1 / 기밀성=Triple-DES
"S/MIME은 Web of Trust로 동작" → ✗ S/MIME = X.509 v3 + CA 필수
"명문-서명 데이터는 내용도 암호화됨" → ✗ 서명만 Base64, 내용은 평문

모든 함정은 PGP·PEM·S/MIME의 특성을 교차로 바꿔치기하는 패턴입니다.

PGP 개발필 짐머만, 1991년, 무료, Web of Trust, CA 없음
PGP 신뢰분산 신뢰 (Web of Trust). X.509 기본 미사용. "인증기관 필요" = ✗
PGP 순서서명 → 압축(ZIP) → 암호화 → Radix-64 (서압암인)
PGP 알고리즘대칭=IDEA/CAST/3DES/AES / 공개키=RSA/ElGamal / 해시=SHA-1/MD5
S/MIME 개발RSA Data Security + IETF. RFC 3850·3851. X.509 v3 인증서 필수. CA 필요.
S/MIME 알고리즘기밀성=Triple-DES / 무결성=SHA-1 / 인증=X.509v3 / 부인방지=DSA
S/MIME 데이터봉함(기밀) / 서명(무결성·인증) / 명문-서명(내용평문) / 서명+봉함(전부)
PEM 개발IETF 표준. 중앙 집중식. Base64 인코딩. 보안능력 PGP보다 우수. 대중화 실패.
3종 비교PGP=분산·CA없음·무료 / S/MIME=중앙·X.509·기업표준 / PEM=중앙·대중화실패
  • "PGP는 CA(인증기관)의 서명이 필요하다" → ✗ (CA 불필요 = PGP 핵심 특징)
  • "PEM은 분산화된 키 인증 방식으로 대중적" → ✗ (중앙집중식, 대중화 어려움)
  • "PGP 처리 순서: 압축 → 서명 → 암호화" → ✗ (서명이 가장 먼저)
  • "S/MIME 무결성 = Triple-DES" → ✗ (무결성=SHA-1 / 기밀성=Triple-DES)
  • "S/MIME = Web of Trust 분산 신뢰" → ✗ (S/MIME = X.509 v3 + CA 중앙집중)
  • "명문-서명 데이터는 내용도 암호화됨" → ✗ (내용은 평문, 서명만 Base64)
  • "PGP = 1991년, 필 짐머만, 무료, Web of Trust, CA 없음" → ✓
  • "PGP 처리 순서: 전자서명 → 압축(ZIP) → 암호화 → Radix-64" → ✓
  • "S/MIME: 기밀성=3DES / 무결성=SHA-1 / 부인방지=DSA / 인증=X.509v3" → ✓
  • "PEM = IETF, 중앙집중식, Base64, 보안능력↑, 대중화✗" → ✓
