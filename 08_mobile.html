1
모바일 운영체제 역사
★★

현재 시장을 양분하는 iOS와 Android는 하루아침에 만들어진 게 아니에요. 모바일 OS의 발전 역사를 간략히 이해하면 각 OS의 보안 철학을 파악하는 데 도움이 돼요.

연도OS / 기기설명
1996년Palm OS개인용 정보 단말기(PDA: Personal Digital Assistance)에서 사용하기 위해 만든 초기 모바일 운영체제.
2000년대Windows CE / Mobile모바일 장치 등에 사용하기 위해 운영체제 Windows CE 기반으로 개발. 현재 거의 사용 안 함.
~2013블랙베리 OSRIM(Research In Motion)에 의해 만들어진 모바일 운영체제. 기업용으로 많이 사용됐으나 현재는 쇠퇴.
2007년~iOS애플이 개발한 모바일 운영체제. Darwin(UNIX에서 발전한 OS) 기반. 폐쇄형·보안 중심. 보안 통제권 = 애플.
2008년~Android여러 핸드폰 제조사들이 참여하여 개발한 개방형 모바일 운영체제. 리눅스 커널(2.6.20)을 모바일 버전으로 만든 운영체제. 보안 통제권 = 개발자 또는 사용자.
2
모바일 보안의 특수성
★★

모바일 기기는 PC 환경과 다른 보안 위협에 노출돼요. ① 물리적 이동성 → 분실·도난 위험이 높아요. ② 항상 연결 상태 → Wi-Fi, 셀룰러, 블루투스 등 다양한 통신으로 공격 경로 증가. ③ 앱 생태계 → 악성 앱이 공식 마켓을 통해 배포될 수 있어요. ④ 개인 정보 집중 → 연락처·위치·금융 정보 등 민감한 데이터가 집중됨. ⑤ 제한된 화면 → 피싱 URL 확인이 어려워 사회공학적 공격에 취약.

1
iOS 보안 특징 — 4대 핵심 요소
★★★ 2016 국가직 출제
iOS 보안 철학

iOS는 애플(Apple)이 보안 통제권을 가지고 있는 폐쇄형 운영체제. Darwin(UNIX에서 발전한 OS) 기반. 모든 앱에 대해 코드 무결성 검사를 수행하여 설치를 제한. 앱스토어를 통해서만 앱 배포 가능(공식 채널 통제).

🔐 ① 안전한 부팅 절차
iOS가 탑재되는 모바일 기기에서 모든 애플리케이션이 암호화된 서명의 확인에 의해 작동한다.
부팅 과정에서 각 단계별로 서명을 검증하여 변조된 소프트웨어가 실행되지 못하도록 방지.
Secure Boot Chain: BootROM → LLB → iBoot → Kernel 순으로 각 단계가 다음 단계의 서명 검증.
✍ ② 응용 프로그램 서명
애플 iOS는 모든 앱에 대해 코드 무결성 검사(Code Signature)를 수행하고 있다.
코드 무결성 검사는 앱에 대한 해시 값으로 서명된 앱 코드 무결성이 다를 경우 실행되지 못하게 한다.
애플의 CA를 통하여 앱에 서명하여 배포. (Android는 개발자가 자체 서명)
📦 ③ 샌드박스(Sandbox) 활용
SandBox = 응용 프로그램이 실행될 때 일종의 가상 머신 안에서 실행되는 것처럼 원래의 운영체제와 완전히 독립되어 실행되는 형태.
사용자의 경우 강제적으로 앱끼리 데이터를 주고받을 수 없고, 시스템 리소스에도 직접 접근할 수 없다.
스크린샷 금지: iOS는 앱이 다른 앱의 화면을 몰래 녹화하는 것을 막는다.
🗑 ④ 원격 삭제 (Remote Wipe)
원격 삭제 기능을 가지고 있어 기기 분실이 있을 경우 연결된 iOS 기기의 데이터를 원격으로 삭제할 수 있다.
Find My(나의 찾기) 서비스를 통해 위치 추적 및 데이터 삭제 가능.

추가 iOS 보안 특징: FTP에서 iOS 로그인 금지 — 서버에서 iOS 기기에 HTTPS와 FTPS를 단순한 파일 전송 기능 이외에는 입력 가능한 응용을 허용하지 않아요. iOS는 Android에 비해 상대적으로 어플리케이션 데이터 접근이 자유롭지 않다고 알려져 있어요.

1
Android 보안 특징
★★★ 2016 국가직 출제
Android 보안 철학

Android는 개발자 또는 사용자에게 보안 통제권이 있는 개방형 운영체제. 리눅스 커널 기반. iOS에 비해 훨씬 자유로운 배포 체계(사이드로딩 허용). 이 때문에 각종 악성코드 감염에 더 취약하다고 알려져 있음.

👤 ① 응용 프로그램 권한 관리
설치된 모든 응용 프로그램은 일반 사용자 권한으로 실행된다.
모든 프로그램 설치 시 권한을 요청하고, 사용자가 접근 권한을 허가받게 하고 있다. 모든 응용 프로그램은 설치 시 사용자 ID를 부여받아 관리한다.
런타임 권한(Android 6.0+): 앱 실행 중 민감한 권한에 대해 사용자에게 동적으로 요청.
✍ ② 응용 프로그램 서명
Android 역시 마찬가지로 응용 프로그램에 대해 서명하고 있다.
iOS의 CA를 통해 서명하여 배포하는 반면, Android는 개발자가 자체 서명하도록 허용하는 점이 가장 큰 차이점이다.
공인 CA 불필요 → 개발자가 직접 자가 서명 인증서 생성 가능. 구글 플레이 스토어 심사는 있지만 iOS보다 느슨함.
📦 ③ 샌드박스 활용
Android 앱 역시 iOS와 마찬가지로 샌드박스 프로세스 내에서 실행되며, 강제적으로 다른 앱에 접근하는 것을 막고 있다.
특정 권한을 갖추어 접근하는 것을 허용하고 있다.
iOS에 비해 상대적으로 어플리케이션 데이터 접근이 더 자유롭다.
⚠ ④ 보안 취약성
Android는 iOS보다 훨씬 자유로운 배포 체계로, 도는 iOS가 하나로 통제되어 있는 반면 Android는 상당히 다양한 버전이 존재한다.
이런 이유로 각종 악성코드에 감염되었으며 이로 인해 피해가 발생하고 있다.
자유로운 앱 배포가 가능한 동시에 iOS에 비해 상대적으로 안전하지 않다고 알려져 있다.

OWASP Mobile Top 10 2016 — M1~M10
★★★
OWASP Mobile Top 10 개요

OWASP(Open Web Application Security Project)는 2001년에 설립된 웹 및 모바일 애플리케이션 보안에 관한 방법론·문서·도구를 만들고 관련 기술을 연구하는 개발자 커뮤니티. 모바일 Top 10은 개발자 커뮤니티 안에서 모바일 앱의 새로운 보안 위협에 대한 인식을 높이기 위해 지속적으로 업데이트됨.

M1
적절하지 않은 플랫폼 사용
플랫폼 기능의 오용 또는 잘못된 플랫폼 보안 설정에 대해 다룸. Android intent, 플랫폼 권한, TouchID 오용, 키 체인 또는 모바일 OS 일부인 다른 보안 컨트롤이 포함될 수 있음.
M2
취약한 데이터 저장소
Mobile Top 10 2014의 M2(취약한 데이터 저장소)와 M4(의도하지 않은 데이터 노출)를 통합. 안전하지 않은 데이터 저장 및 의도하지 않은 데이터 유출에 대해 다룸.
M3
취약한 통신
악의적인 핸드셰이킹, 잘못된 SSL 버전, 약한 협상, 민감정보의 평문 통신 등을 다룸.
M4
취약한 인증
최종 사용자 인증 또는 잘못된 세션 관리의 개념을 포함. ①사용자 식별을 못하는 경우 ②사용자 신원이 유지되지 않는 경우 ③세션 관리 취약점.
M5
취약한 암호화
중요한 정보 자산에 암호화를 적용하고 올바르지 않은 암호화를 포함. M3(취약한 통신)에서 TLS 또는 SSL과 관련된 사항들이 있음. 앱이 암호화 사용을 해야 할 때 암호화하지 않았다면 M2에 속함.
M6
취약한 권한부여
잘못된 권한(Authorization) 부여(예: 클라이언트 측의 권한 부여, 강제 검색 등)를 다룸. 이는 인증(Authentication) 문제(예: 기기 등록, 사용자 식별 등)와 구별됨.
M7
취약한 코드 품질
신뢰할 수 없는 입력을 통한 보안 결정. 모바일 클라이언트의 코드 수준 구현 문제. 서버 측 코딩 실수와 구별됨. 버퍼 오버플로, 포맷스트링 취약점, 모바일 장치에서 실행되는 일부 코드를 재작성해야 하는 코드 레벨에서 실수하는 것들.
M8
코드 변조
바이너리 패치, 로컬 리소스 수정, 메서드 후킹, 메서드 변경, 동적 메모리 수정이 포함됨. 공격자는 코드를 직접 수정하거나 메모리 내용을 동적으로 변경하거나 앱이 사용하는 시스템 API를 변경·대체하거나 앱의 데이터와 자원을 수정할 수 있음.
M9
역공학(Reverse Engineering)
공격자가 앱의 바이너리를 분석하여 소스 코드·라이브러리·알고리즘·기타 자산을 복원. 난독화가 적용되지 않은 앱은 디컴파일을 통해 내부 로직 노출.
M10
불필요한 기능
개발자가 숨겨진 백도어 기능 또는 내부 개발 보안 컨트롤이 포함된 채로 앱을 출시하는 경우. 테스트 코드, 관리자 계정, 로그 정보 등이 프로덕션 앱에 그대로 남아 있는 경우.
시험 포인트: M1=플랫폼 오용 / M2=취약한 데이터 저장 / M3=취약한 통신 / M4=취약한 인증 / M5=취약한 암호화 / M6=취약한 권한부여 / M7=취약한 코드품질 / M8=코드변조 / M9=역공학 / M10=불필요한 기능. M4(인증)와 M6(권한부여) 혼동 주의.
1
모바일 위협 및 공격 유형
★★★
스미싱 (SMishing)
SMS + Phishing. 문자 메시지를 이용한 피싱 공격. 악성 URL 포함 문자 전송 → 클릭 시 악성 앱 설치 또는 개인정보 탈취. 은행·택배·공공기관 사칭.
피싱 (Phishing)
이메일·가짜 웹사이트로 정상적인 사이트처럼 위장. 로그인 정보·카드번호 등 개인정보 탈취. 모바일에서는 URL 확인이 어려워 더 취약.
파밍 (Pharming)
DNS 변조로 정상 URL을 입력해도 악성 사이트로 유도. 사용자가 인식하지 못하는 상태에서 개인정보 탈취. 호스트 파일 변조도 포함.
루팅 / 탈옥
Android 루팅(Rooting): 시스템 최고 권한(root) 획득. iOS 탈옥(Jailbreak): 애플의 제한 해제. 양쪽 모두 기본적인 보안 경계가 무너져 악성코드 설치 위험 증가.
악성 앱
공식 마켓 우회 또는 마켓 내 악성 앱. 트로이목마 형태로 정상 앱처럼 위장. 설치 후 연락처·문자·위치정보 탈취, 원격 제어 수행.
무선 도청
공개 Wi-Fi에서 스니핑으로 암호화 안 된 통신 도청. Evil Twin(가짜 AP) 접속 유도 → 모든 통신 감청. MITM(중간자) 공격.
2
BYOD (Bring Your Own Device)
★★
BYOD 정의

BYOD(Bring Your Own Device)는 개인 모바일 기기를 업무에 사용하는 것. 기업의 효율성·편의성을 제공하지만 직원 기기에 대한 보안 관리가 필요. 개인 기기에 기업 데이터가 혼재하여 분실·도난 시 정보유출 위험 증가.

구분장점단점/위협
BYOD 특징 업무 효율성·편의성 향상. 직원 만족도 증가. 기기 구매 비용 절감. 분실·도난 시 기업 데이터 유출. 악성 앱 설치 가능. 보안 정책 적용 어려움. 개인 앱과 기업 앱 혼재.
MDM 대응 MDM(Mobile Device Management): 기업이 직원 기기를 원격으로 관리. 보안 정책 강제 적용. 직원 개인 정보 감시 논란. 개인 기기 제어 범위 논쟁.

④비밀번호 설정 및 정기 변경분실 또는 도난 시 정보 유출·악성코드 설치 위험을 위해 단말기 비밀번호 설정 필요.

⑥증상 지속 시 악성코드 여부 확인이상 증상 발생 후 조치해도 지속되면 악성코드 감염 의심 → 백신으로 진단·치료 필요.
⑦바이러스 백신 설치핸드폰용 특정 폴더나 파일에 숨겨져 있을 수 있으므로 백신으로 바이러스 유무 검사.
⑧정기적으로 바이러스 검사PC와 모바일 데이터 백업·복사·전송 수행 시 악성코드가 있을 수 있으므로 정기 점검.

⑩운영체제 최신 버전 유지취약점이 보안 위협에 이용될 수 있으므로 항상 최신 버전으로 업데이트하여 사용.

📄 2016 국가직 9급 정보시스템보안 문1 · iOS vs Android 오답 찾기★★★
모바일 운영체제인 iOS와 안드로이드의 보안 체계에 대한 설명으로 옳지 않은 것은?
① iOS는 모든 앱에 대한 코드 무결성 점검을 수행하여 설치를 제한한다.
② iOS와 안드로이드 모두 프로그램의 실행 권한이 일반 사용자에게 있다.
③ iOS는 애플의 CA를 통하여 앱을 서명 및 배포하고, 안드로이드는 개발자가 서명 및 배포한다.
④ 보안 통제권이 iOS는 애플에 있고, 안드로이드는 개발자나 사용자에게 있다.
정답 ②

② iOS는 프로그램 실행 권한이 애플(시스템)에 의해 엄격히 제어돼요. "일반 사용자 권한으로 실행"은 Android의 특징. iOS의 프로그램 권한은 시스템(애플)이 관리.

①iOS 코드 무결성 점검 → 맞음. ③iOS=CA 서명, Android=개발자 자체 서명 → 맞음. ④iOS=애플 통제, Android=개발자·사용자 통제 → 맞음.
📄 경찰공무원 기출 · 스미싱·피싱·파밍 개념 구분★★★
컴퓨터 기반 사회공학적 공격기법에 해당하지 않는 것은?
① 피싱(phishing)
② 파밍(pharming)
③ 스미싱(smishing)
④ 스푸핑(spoofing)
정답 ④ — 스푸핑(Spoofing)은 기술적 공격(IP·MAC·DNS 주소 위조)으로 사회공학적 기법이 아니에요.

· 피싱 = 이메일·가짜 사이트로 정보 탈취 → 사회공학적.
· 파밍 = DNS 변조로 가짜 사이트 유도 → 기술+사회공학 복합.
· 스미싱 = SMS로 피싱 → 사회공학적.
· 스푸핑 = IP·MAC 주소 위조 → 순수 기술적 공격.
