1
이메일의 탄생 — SMTP의 순수한 시절
1970~80년대

인터넷 초창기, 이메일은 그냥 텍스트 엽서였어요. SMTP(Simple Mail Transfer Protocol)는 말 그대로 "심플"합니다. 서버에서 서버로 텍스트를 전달하는 우체국 역할이에요.

문제는 이게 정말 순진하게 설계됐다는 거예요.

"보내는 사람이 admin@회사.com 이라고 쓰면, 그냥 admin@회사.com이 보낸 걸로 취급"
→ 발신자 인증 없음. 내용 암호화 없음. 중간에 누가 봐도 모름.

1970년대에 인터넷을 쓰는 사람들은 서로 아는 연구자들이었으니까, 그냥 신뢰 기반으로 만든 거였죠. 이게 나중에 거대한 구멍이 됩니다.

2
MIME의 등장 — "텍스트만 보내기 싫어요"
1990년대 초

그런데 시간이 지나면서 문제가 생겼어요. SMTP는 7비트 ASCII 텍스트만 전송할 수 있게 설계됐어요. 사진을 보내고 싶어요? 한글을 쓰고 싶어요? 첨부파일을 넣고 싶어요? SMTP만으로는 안 돼요.

그래서 나온 게 MIME(Multipurpose Internet Mail Extensions)입니다.
"어차피 SMTP는 텍스트만 보내니까, 사진·파일 같은 이진 데이터를 텍스트로 인코딩해서 보내자"

이게 Base64 인코딩입니다. 이진 파일을 A-Z, a-z, 0-9, +, / 문자로 바꿔서 텍스트인 척 전송하는 거예요. 이메일 클라이언트가 받아서 다시 복원하는 방식이죠.

MIME 덕분에 이메일로 사진도, 한국어도, PDF도 보낼 수 있게 됐어요. 하지만 보안 문제는 여전히 해결되지 않았어요.

3
이메일은 여전히 "엽서" — PGP의 탄생
1991년, 필 짐머만

MIME으로 다양한 파일을 보낼 수 있게 됐는데, 여전히 본질적인 문제가 남아 있었어요.

이메일은 인터넷을 거치는 동안 누구나 읽을 수 있는 엽서
서버에서 서버로 넘어갈 때마다 평문으로 돌아다녀요. 관리자가 마음먹으면 내용을 다 볼 수 있어요.

1991년, 필 짐머만(Phil Zimmermann)이라는 프로그래머가 이걸 보다 못해서 직접 만들어 공개한 게 PGP(Pretty Good Privacy)예요.

💡 PGP가 해결한 핵심 문제 ① — "어떻게 암호화하나"

내가 너에게 암호화된 편지를 보내려면, 너의 공개키로 편지를 잠가야 해요. 그런데 공개키 암호화는 느려요. 긴 편지 전체를 RSA로 암호화하면 시간이 오래 걸려요. PGP의 해결책은 이거였어요.

문제: 공개키 암호화는 느리다
해결: 실제 편지 → 빠른 대칭키(IDEA 등)로 암호화
대칭키 → 상대방의 공개키로 암호화해서 함께 전송

→ 이게 하이브리드 암호화. 오늘날 HTTPS도 똑같은 방식!
💡 PGP가 해결한 핵심 문제 ② — "누가 공개키를 검증하나"

공개키 암호화의 또 다른 문제가 있어요. 네이버라고 주장하는 서버의 공개키가 진짜 네이버 것인지 어떻게 알아요?

S/MIME과 HTTPS는 이걸 인증기관(CA)이 해결해요. CA가 "이 공개키는 진짜 네이버 거야"라고 인증서에 서명해주는 중앙 집중식 신뢰 모델이죠.

PGP는 달랐어요. "CA 같은 중앙 기관을 믿지 말자. 내 친구가 그 사람 공개키를 믿는다면, 나도 믿겠다" — 이게 Web of Trust(신뢰의 거미줄). 중앙 서버 없이 사람들 사이의 신뢰가 퍼져나가는 분산형 모델이에요.

4
S/MIME — "기업은 PGP가 불편해요"
1995년, RSA 사

PGP는 철학은 좋은데 실용적인 문제가 있었어요.

Web of Trust가 현실에서 작동하려면, 내가 이메일 보내는 모든 사람과 미리 공개키를 교환해야 해요. 기업 환경에서는 너무 불편해요. 처음 연락하는 거래처에 이메일 보내기 전에 공개키부터 교환하자고 할 수는 없잖아요.

PGP — 분산 신뢰

인증기관(CA) 없음. 사용자끼리 직접 공개키 교환·서명. 자유롭지만 번거로움. X.509 인증서 미사용.

S/MIME — 중앙 신뢰

CA가 발급한 X.509 v3 인증서 사용. 처음 보는 상대도 바로 암호화 통신 가능. 기업 환경에 적합.

기업에서 사용하는 이메일 암호화는 대부분 S/MIME 방식이에요. Outlook이나 Gmail 기업용에서 이메일 암호화를 켜면 S/MIME을 씁니다.

5
웹의 폭발적 성장 — HTTP의 똑같은 문제
1990년대 초

1990년대 초, 팀 버너스리가 WWW를 만들고 HTTP를 설계했어요. 그리고 이메일과 똑같은 문제가 발생했어요.

"인터넷뱅킹을 하는데 비밀번호가 평문으로 날아다닌다"
HTTP는 설계상 보안을 고려하지 않았어요. SMTP와 똑같은 실수.

1994년 넷스케이프에서 이걸 해결하려고 나섰어요. 여기서 두 가지 다른 접근이 경쟁했습니다.

6
S-HTTP vs HTTPS — 두 가지 접근의 대결
1994년, 결정적 분기점

S-HTTP (패자)

HTTP 자체를 보안 버전으로 바꾸자. HTTP 프로토콜 안에 암호화 기능 내장. 개념적으로 깔끔하지만 기존 HTTP와 호환이 안 됨. 서버도, 클라이언트도 전부 바꿔야 함.

HTTPS (승자)

HTTP는 건드리지 말자. 아래에 암호화 계층 하나를 끼워 넣자. 넷스케이프가 선택한 발상의 전환. 이 "끼워 넣은 계층"이 SSL.

⚡ 왜 HTTPS가 이겼나 — 범용성의 승리

S-HTTP는 HTTP에만 쓸 수 있었어요. 반면 SSL은 어떤 프로토콜 아래에도 끼워 넣을 수 있었어요.

FTP + SSL = FTPS / SMTP + SSL = SMTPS / IMAP + SSL = IMAPS

SSL이 "범용 암호화 레이어" 역할을 한 거예요. S-HTTP는 역사 속으로 사라지고 HTTPS가 표준이 됐습니다.

7
SSL이 HTTP와 TCP 사이에 끼어든 이야기
핵심 구조 이해

TCP/IP 스택을 생각해보면, 원래 HTTP는 TCP 위에서 직접 동작해요.

기존 구조 (HTTP)
┌─────────────────┐
│ 응용층 HTTP │ ← "TCP야, GET /index.html 보내줘"
├─────────────────┤
│ 전송층 TCP │ ← 평문 그대로 전송
└─────────────────┘

새 구조 (HTTPS)
┌─────────────────┐
│ 응용층 HTTP │ ← 아무것도 모름. 여전히 "TCP야~" 라고 함
├─────────────────┤
│ [투명 유리창] SSL │ ← 여기서 몰래 암호화/복호화
├─────────────────┤
│ 전송층 TCP │ ← 암호화된 데이터만 봄
└─────────────────┘
HTTP 입장에서는 아무것도 변한 게 없어요.
"TCP야, GET /index.html 보내줘" → SSL이 중간에서 이걸 암호화 → TCP에 전달
HTTP는 자기가 TCP한테 직접 말하는 줄 알아요. 눈치 못 챕니다.

이 덕분에 어마어마한 장점이 생겼어요. HTTP만 보안이 필요한 게 아니에요. FTP도, SMTP도, IMAP도 모두 SSL을 그냥 끼워 쓸 수 있어요. 계층 구조의 아름다움이 빛을 발한 순간이에요.

SSL은 전송층(TCP)과 응용층(HTTP) 사이의 투명한 보안 유리창
8
SNMP — 네트워크 장비들의 건강 검진
관리자의 고민

조금 다른 이야기로 넘어가서 SNMP예요. 네트워크 규모가 커지면 문제가 생겨요.

"라우터가 100대, 스위치가 200대인데... 어디서 문제가 났는지 어떻게 알아요?"
각 장비에 직접 들어가서 확인할 수 없잖아요.

그래서 만든 프로토콜이 SNMP(Simple Network Management Protocol)예요. 구조가 재미있어요. 의사와 환자 관계랑 비슷해요.

NMS (Network Management System) = 의사
→ 모든 환자(장비) 상태를 모니터링하는 중앙 관리 시스템

Agent = 각 장비에 설치된 주치의
→ 자기 장비 상태를 실시간으로 체크하고 기록

MIB (Management Information Base) = 의료 차트
→ Agent가 관리하는 데이터 목록
→ "인터페이스 트래픽은 얼마나 됩니까? CPU는요?"
💡 평상시 vs 응급 상황 — GetRequest vs Trap

평상시: NMS가 Agent한테 주기적으로 물어봐요. "너 건강 상태 보고해봐" (GetRequest) → Agent가 답해요. "현재 CPU 35%, 트래픽 1Gbps" (GetResponse)

응급 상황: 갑자기 장비가 과부하 상태가 됐어요. Agent가 기다리지 않고 NMS한테 먼저 연락해요. "저 지금 이상해요!" — 이게 Trap. Agent가 NMS에게 능동적으로 보내는 응급 알람.

SNMP가 UDP를 쓰는 이유도 명확해요. 장비가 수백 대인데 모든 장비와 TCP 연결을 유지하면 부담이 너무 크거든요. UDP로 "질문-답변"만 빠르게 주고받는 게 훨씬 효율적이에요.

📌 전체 흐름 한 줄 요약

이 프로토콜들의 역사는 사실 하나의 공통된 스토리예요.


"편리하게 만든 것들이 성공했지만, 보안을 나중에 생각한 탓에 구멍이 생겼다. 그 구멍을 막으려는 시도들이 프로토콜 역사다"


SMTP는 텍스트 전달에 집중 → MIME으로 확장 → PGP/S/MIME으로 보안

HTTP는 문서 전달에 집중 → S-HTTP(실패) / HTTPS(성공)로 보안

SNMP는 네트워크 관리에 집중 → SNMPv3으로 인증·암호화 추가

1
웹(WWW) 보안 개요
기초
WWW 정의

WWW(World Wide Web)은 하이퍼텍스트(hyper-text) 문서들이 링크를 통해 연결된 전 세계 인터넷 정보 공간이다. 웹 브라우저를 통해 다양한 정보에 손쉽게 접근할 수 있으나, 보안에서 검증되지 않은 일반 텍스트 기반 프로토콜(HTTP)이 자주 이용되어 보안 위협이 발생한다.

웹 보안이 필요한 이유
HTTP는 평문 전송 → 도청·변조 취약. 전자상거래·인터넷뱅킹 등 민감 정보 전송 증가. 웹 서버·브라우저·호스트 모두 안전한 통신 필요.
HTTP 기본 특성
HyperText Transfer Protocol. TCP 80번 포트. HTTP/3부터 UDP 사용. Stateless(상태 저장 안 함). 요청/응답(Request/Response) 프로토콜.
보안 요구사항
기밀성 / 메시지 인증 / 사용자 인증 / 송수신 부인 방지 / 메시지 재생 방지
OWASP
Open Web Application Security Project. 10대 웹 취약점 발표. SQL Injection, XSS, CSRF 등 포함.

4
SSL/TLS 구조 — 4가지 프로토콜
★★★
SSL/TLS 정의

SSL(Secure Socket Layer)은 넷스케이프(Netscape)가 1994년 처음 제안한 전송 계층 보안 프로토콜. TCP/IP 계층과 응용 계층 사이에 위치하여 종단 간 보안 서비스를 제공. IETF 인터넷 표준으로 TLS(Transport Layer Security)로 발전. 단 하나의 프로토콜이 아닌 두 계층의 프로토콜로 구성됨.

SSL/TLS 프로토콜 스택 구조 — 4가지 프로토콜
프로토콜역할핵심 내용
① Handshake Protocol 인증·암호 협상 서버-클라이언트 상호 인증. MAC 알고리즘·암호화 방식·세션 키 협상. 모든 응용 데이터 전송 이전에 수행.
② Change Cipher Spec 암호 사양 전환 핸드셰이크에서 협의된 압축·MAC·암호화 알고리즘을 실제 적용하기 시작함을 알림. 가장 단순한 SSL 프로토콜(1바이트).
③ Alert Protocol 오류·경고 알림 SSL 오류를 상대방에게 전달. 경고(warning) 또는 치명적(fatal) 두 수준. fatal이면 즉시 연결 종료.
④ Record Protocol 기본 보안 서비스 기밀성(대칭키 암호화) + 메시지 무결성(MAC) 제공. 상위 프로토콜 데이터 처리. HTTP는 SSL Record 위에서 동작.
⚠함정 — "user authentication 프로토콜이 SSL 구성요소" (2016 정보시스템보안)
④ "user authentication 프로토콜" → ✗ 오답 (존재하지 않는 SSL 프로토콜)

SSL 4가지 구성: Handshake / Change Cipher Spec / Alert / Record. "user authentication 프로토콜"은 SSL의 구성 프로토콜이 아님. CCS(Change Cipher Spec)와 Alert도 반드시 암기.

5
SSL Record 프로토콜 처리 순서
★★★ 매년 출제!
핵심: SSL Record 프로토콜은 상위 계층 메시지를 5단계로 처리. 단편화 → 압축 → MAC 추가 → 암호화 → 헤더 추가 순서가 매년 기출.
SSL Record 프로토콜 처리 5단계 — 송신측 기준
핵심 암기 순서: 단 압 MAC 암 헤 — 단편화 → 압축 → MAC 추가 → 암호화 → SSL 레코드 헤더 붙이기. MAC이 암호화보다 먼저임을 주의!
단계명칭상세 설명
① 단편화Fragmentation상위 계층 데이터를 최대 16,384 bytes(2^14) 이하 블록으로 분할
② 압축Compression손실 없는(lossless) 압축 알고리즘 적용. 데이터 팽창 없어야 함
③ MAC 추가MAC Generation핸드셰이크에서 협의된 HMAC으로 무결성 검증값 생성·첨부
④ 암호화Encryption압축된 데이터 + MAC을 대칭키(세션키)로 암호화
⑤ 헤더 추가Header Attach콘텐츠 유형(8bit) + 버전(16bit) + 압축 길이(16bit) 헤더 추가 후 전송
⚠Record 처리 순서 함정 — 2016·2021·2023 국가직·서울시 반복 출제
④ 단편화 → 암호화 → MAC 첨부 → 압축 → 헤더 → ✗ 오답
① 압축 → 단편화 → 암호화 → MAC 첨부 → 헤더 → ✗ 오답 (2016 국가직 오답 선지)

핵심 함정 2개: ① 압축이 단편화보다 먼저 오는 선지 → ✗ (단편화가 먼저). ② 암호화가 MAC보다 먼저 오는 선지 → ✗ (MAC이 먼저). 정답은 항상 단편화 → 압축 → MAC → 암호화 → 헤더.
