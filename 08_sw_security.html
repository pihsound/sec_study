왜 만든 후에 고치는 게 아니라, 만들면서 보안을 넣어야 할까?

소프트웨어 보안에는 유명한 법칙이 있어요. "설계 단계에서 보안 결함을 고치는 비용이 1이라면, 배포 후 고치는 비용은 100배"라는 거예요. 완성된 건물에서 기둥을 바꾸는 게 설계도 단계에서 바꾸는 것보다 훨씬 어렵고 비싼 것과 같아요.

과거에는 개발팀과 보안팀이 따로 움직였어요. 개발팀이 소프트웨어를 완성하면, 보안팀이 마지막에 "침투 테스트"를 해서 취약점을 발견하는 방식이었어요. 그런데 이 방식에는 치명적인 문제가 있었어요. 취약점이 발견됐을 때는 이미 구조를 바꾸기 너무 늦은 시점이에요. 고치려면 엄청난 비용과 시간이 들거나, 그냥 출시하는 경우도 많았어요.

2002년, 마이크로소프트는 이 문제를 정면으로 직시했어요. Windows XP가 출시된 직후, MS 소프트웨어에서 보안 취약점이 쏟아지기 시작했고 엄청난 비판을 받았어요. 빌 게이츠가 직접 "신뢰할 수 있는 컴퓨팅(Trustworthy Computing)" 메시지를 전사에 보냈어요. 그리고 SDL(Secure Development Lifecycle)이라는 체계가 탄생했어요. "보안을 나중에 붙이는 게 아니라 개발 전 과정에 녹여 넣자."

SDL이 적용된 소프트웨어는 이전에 비해 취약점이 50% 이상 감소했어요. 이후 Seven Touchpoints, CLASP 같은 다양한 소프트웨어 보안 방법론이 나왔어요. 그리고 이런 접근의 핵심 실천 활동이 시큐어코딩(Secure Coding)이에요. 소스코드에 잠재된 보안 취약점을 처음부터 제거하면서 개발하는 방식이에요.

소프트웨어 보안의 핵심 원칙:
"보안은 나중에 붙이는 게 아니라 개발 전 과정에 녹여 넣어야 한다."
설계 → 구현 → 테스트 → 배포 모든 단계에서 보안을 고려하는 것 = SDLC 보안 (Security in SDLC)
1
시큐어코딩 (Secure Coding)
★★★ 정의 암기
시큐어코딩 정의

시큐어코딩(Secure Coding)이란 안전한 소프트웨어 개발을 위해 소스코드 등에 존재할 수 있는 잠재적인 보안 취약점을 제거하고, 보안을 고려하여 기능을 설계 및 구현하는 등 소프트웨어 개발 과정에서 지켜야 할 보안 활동이다. 단순히 코딩 규칙을 지키는 것을 넘어, 개발 전 단계에서 보안을 내재화하는 접근 방식.

시큐어코딩의 핵심 키워드를 기억해야 해요. "소스코드의 잠재적인 보안 취약점 제거", "개발 과정에서 지켜야 할 보안 활동"이에요. 시험에서 "스캐빈징(Scavenging)", "살라미(Salami)", "웨어하우스" 등과 함께 혼동하도록 선지에 나와요.

⚠함정 — 시큐어코딩 정의 혼동 (2017 서울시 A·B형 동일 출제)
"안전한 소프트웨어 개발을 위해 소스코드 등에 존재할 수 있는 잠재적인 보안 취약점을 제거하고, 보안을 고려하여 기능을 설계 및 구현하는 등 소프트웨어 개발 과정에서 지켜야 할 보안 활동" → ✓ 시큐어코딩

혼동 유발 선지: 스캐빈징(Scavenging) = 저장 매체에서 지워진 데이터를 복구하는 행위. 살라미(Salami) = 소액 반복 사기. 웨어하우스 = 데이터 창고. 전혀 다른 개념이에요.

2
소프트웨어 개발 보안 3대 방법론
★★★
MS-SDL
Microsoft Secure Development Lifecycle. MS가 보안 취약점 감소를 위해 도입. SDL 적용 후 이전 대비 취약점 50% 이상 감소. 소프트웨어 개발 전 단계(요구사항→설계→구현→테스트→배포)에 보안 활동 통합.
Seven Touchpoints
실용적 소프트웨어 보안을 위한 7가지 모범 사례. 공통 취약점을 파악하고 해결하며, 하고 외부 소프트웨어에 대해 리뷰하고 객관적인 분석 및 위험 분석을 거쳐 안전한 제품을 만드는 방법. 개발 동안 소프트웨어 보안을 위험에 따라 관리.
CLASP
Comprehensive, Lightweight Application Security Process. 활동 중심 프로세스 (Activity-Driven Process). 소프트웨어 개발 초기 단계에 보안을 통합. 이미 운영 중인 시스템에 적용하기 좋음. 역할·활동·평가·취약점으로 구성.

소프트웨어 보안 약점 7가지 유형
★★★ 유형·설명 교차 출제

소프트웨어 보안 약점은 행정안전부·KISA가 정한 7가지 유형으로 분류돼요. 시험에서는 유형 이름과 설명이 바뀌어 나오거나, 어느 유형에 속하는지 묻는 형태가 많아요. 특히 보안 기능 ↔ 코드 오류 ↔ API 오용의 구분이 핵심 함정 포인트예요.

① 입력 데이터 검증 및 표현
정의: 프로그램 입력값에 대한 검증 누락 또는 부적절한 검증, 데이터의 잘못된 형식 지정으로 인해 발생.
예시: SQL 삽입, XSS, XQuery 삽입, LDAP 삽입, 운영체제 명령어 삽입, XML 삽입, 버퍼 오버플로, 포맷 스트링
② 보안 기능
정의: 인증, 접근제어, 암호화, 권한 관리 등의 기능을 부적절하게 구현할 때 발생.
예시: 부적절한 인가, 하드코드된 암호화 키, 취약한 암호화 알고리즘 사용, 적절하지 않은 난수 생성
③ 시간 및 상태
정의: 동시 또는 거의 동시 수행을 지원하는 병렬 시스템·프로세스 간 자원(시간, 상태) 관리 미흡.
예시: 레이스 컨디션(TOCTOU), 종료되지 않은 반복문/재귀 함수
④ 에러 처리
정의: 에러를 처리하지 않거나, 불필요한 정보를 에러 메시지로 외부에 노출.
예시: 오류 메시지에 시스템 정보 포함, 부적절한 예외 처리, 오류 상황 무시
⑤ 코드 오류
정의: 개발자가 실수로 발생시킨 코드 오류. 의도하지 않은 방법으로 API를 사용하거나 보안에 취약한 API를 사용.
예시: DNS lookup에 의존한 보안 결정, 잘못된 형변환, Null 포인터 역참조
⑥ 캡슐화
정의: 중요한 데이터 또는 기능성을 불충분하게 캡슐화하였을 때, 인가되지 않은 사용자에게 데이터 누출 가능.
예시: 잘못된 세션에 의한 데이터 노출, 시스템 데이터 정보 노출, Public 메서드로 민감정보 접근 허용
⑦ API 오용
정의: 의도된 사용에 반하는 방법으로 API를 사용하거나, 보안에 취약한 API를 사용. ⑤ 코드 오류와 혼동 주의.
예시: DNS lookup에 의존한 보안 결정(→사실 코드 오류 예시), Socket 통신에서 취약한 함수 사용, 취약한 암호화 API 직접 호출
⚠함정 ★★★ — 보안 약점 유형 설명 교차 (2021 서울시 7급)
선지 ①: "코드 오류 — 의도하지 않은 방법으로 API를 사용하거나 보안에 취약한 API를 사용 (예: DNS lookup에 의존한 보안결정)" → ✗ 오답

DNS lookup에 의존한 보안결정은 코드 오류의 예시이지만, 이 선지의 설명 자체는 API 오용의 설명이에요. 교재마다 경계가 모호한 부분이어서 시험에 자주 출제돼요. ①은 오답이에요.

선지 ④: "보안 기능 — 인증, 접근제어, 암호화 등의 기능을 부적절하게 구현 시 발생 (예: 부적절한 인가, 하드코드된 암호화키)" → ✓ 정답

보안 기능 유형의 정확한 설명. 인증·접근제어·암호화 = 보안 기능. 부적절한 인가·하드코드된 암호화키 = 보안 기능 약점 예시.

7가지 유형을 빠르게 구분하는 핵심 키워드를 정리하면, ①입력검증=SQL삽입·XSS, ②보안기능=인증·접근제어·암호화 부적절 구현, ③시간상태=레이스컨디션, ④에러처리=오류 메시지 정보 노출, ⑤코드오류=잘못된 형변환·Null 역참조, ⑥캡슐화=세션 데이터 노출, ⑦API오용=취약한 API 사용이에요.

OWASP 개요
★★★
OWASP 정의

OWASP(Open Web Application Security Project)는 2001년에 설립된 웹 및 모바일 애플리케이션 보안에 관한 방법론, 문서, 도구를 만들고 관련 기술을 연구하는 개발자 커뮤니티. 상위 10개 위험 목록(TOP 10)을 개발자 커뮤니티 안에서 주기적으로 업데이트. 비영리 단체.

2
OWASP TOP 10 연도별 변화 — 완전 정리
★★★ 연도별 차이 출제

OWASP TOP 10은 가장 자주 출제되는 주제예요. 단순 암기보다 연도별 변화 트렌드를 이해해야 해요. 특히 2013→2017/2020→2021/2022의 A1·A3 위치 변화, 새로 등장한 항목(XXE·SSRF), 사라진 항목(CSRF·Unvalidated Redirects)이 시험에 자주 나와요.

핵심 변화 포인트 3가지:

① 2013년 A1=인젝션이었다가 2021년에는 A1=취약한 접근제어로 밀려났어요. 인젝션은 A3로 내려갔어요.

② 2017년에 XXE(XML 외부개체)가 A4로 새로 등장했어요. 2013년까지는 없던 항목이에요.

③ 2021년에 SSRF(서버사이드 요청 위조)가 A10으로 새로 등장했어요. CSRF는 2017년부터 이미 TOP 10에서 사라졌고, 2021에도 없어요.

CVE · CWE · CVSS — 취약점 표준 체계
★★★ CVE 형식 출제
CVE 정의

CVE(Common Vulnerabilities and Exposures)는 공개적으로 알려진 소프트웨어의 보안 취약점을 가리키는 고유 표기. 정보 보안 취약점의 표준 코드. 장치·시스템·프로그램 해킹에 악용될 수 있는 컴퓨터 보안 취약 시스템에 부여됨. CVE 형식: CVE-[취약점이 발견된 연도]-[일련번호]. 예: CVE-2022-34993.

CVE 형식은 시험에 자주 나와요. CVE-[연도]-[일련번호] 순서를 반드시 외워야 해요. 2023 국가직에서 "(가) 연도, (나) 일련번호"가 정답으로 출제됐어요.

표준Full Name정의시험 키워드
CVE Common Vulnerabilities and Exposures 공개적으로 알려진 보안 취약점의 고유 식별 코드. 형식: CVE-[연도]-[일련번호]. MITRE에서 관리. ★ 연도-일련번호 순서
CWE Common Weakness Enumeration 소프트웨어 보안 약점 유형 목록. CVE가 특정 취약점 인스턴스라면, CWE는 취약점 패턴/유형. 예: CWE-79=XSS. 취약점 유형(패턴)
CVSS Common Vulnerability Scoring System 취약점의 심각도를 0~10 점수로 평가하는 표준. 실제 발생한 취약점 등급 매기는 공개 프레임워크. 2024 국가직 기출. 취약점 심각도 점수
NVD National Vulnerability Database 미국 NIST가 운영하는 취약점 데이터베이스. CVE 기반 취약점 정보 검색 가능. NIST, 취약점 DB
⚠함정 ★★★ — CVE vs CVSS vs CWE 구분 (2023·2024 국가직)
CVE 취약점명 형식: "CVE-[연도]-[일련번호]" → (가)=연도, (나)=일련번호 → ✓ 정답 ④ (2023 국가직)

예: CVE-2022-34993 → 2022년 발견, 일련번호 34993. 연도가 앞, 번호가 뒤.

"보안 취약점으로 인해 실제 발생한 취약점의 등급을 매기는 공개 프레임워크" → ✓ CVSS (2024 국가직)

CVE = 취약점 식별. CVSS = 취약점 심각도 점수. CWE = 취약점 유형 분류. NVD = 취약점 데이터베이스.

연도순
📄 2016 국가직 9급 · OWASP 2013 상위 3개 아닌 것★★★
OWASP에서 발표한 2013년도 10대 웹 애플리케이션 보안 위험 중 발생 빈도가 높은 상위 3개에 속하지 않는 것은?
① Injection
② Cross-Site Scripting
③ Unvalidated Redirects and Forwards
④ Broken Authentication and Session Management
정답 ③ — 2013년 TOP3: A1=Injection, A2=인증 및 세션 관리 취약점, A3=XSS. Unvalidated Redirects and Forwards는 A10(맨 아래). 상위 3개에 속하지 않아요.
📄 2017 서울시 A·B형 · 시큐어코딩 정의★★★
다음의 지문은 무엇을 설명한 것인가?
"안전한 소프트웨어 개발을 위해 소스코드 등에 존재할 수 있는 잠재적인 보안 취약점을 제거하고, 보안을 고려하여 기능을 설계 및 구현하는 등 소프트웨어 개발 과정에서 지켜야 할 보안 활동이다."
① 시큐어코딩(Secure Coding)
② 스캐빈징(Scavenging)
③ 웨어하우스(Warehouse)
④ 살라미(Salami)
정답 ① — 키워드: "소스코드의 잠재적인 보안 취약점 제거", "개발 과정에서 지켜야 할 보안 활동" = 시큐어코딩. 스캐빈징=저장매체 데이터 복구. 살라미=소액 반복 사기. A·B형 모두 동일하게 출제됐어요.
📄 2017 국회직 9급 · OWASP 2013 속하지 않는 것★★★
OWASP에서 2013년에 발표한 10대 웹 취약점에 속하지 않는 것은?
① 인젝션
② 크로스 사이트 요청 변조 (CSRF)
③ 취약한 간접 객체 참조
④ 검증되지 않은 리다이렉트 및 포워드
정답 ③ — 2013 TOP 10에는 취약한 직접 객체 참조(Insecure Direct Object Reference, A4)가 있어요. "간접 객체 참조"는 OWASP TOP 10에 없는 용어. CSRF는 A8로 포함됨.
📄 2020 국회직 9급 · OWASP 2020 속하지 않는 것★★★
OWASP(Open Web Application Security Project) 2020에서 발표된 10가지 보안 위협에 속하지 않는 것은?
① Abuse of Cloud Computing
② Injection
③ XML External Entities
④ Broken Authentication
⑤ Sensitive Data Exposure
정답 ① — Abuse of Cloud Computing(클라우드 컴퓨팅 남용)은 OWASP TOP 10에 포함된 적 없는 항목. 2017/2020 TOP 10: A1=Injection, A4=XXE, A2=Broken Authentication, A3=Sensitive Data Exposure 모두 포함돼 있어요.
📄 2021 서울시 7급 · 소프트웨어 보안 약점 유형 옳은 것★★★
소프트웨어 보안 약점 유형에 대한 설명으로 가장 옳은 것은?
① 코드 오류: 의도하지 않은 방법으로 API를 사용하거나 보안에 취약한 API를 사용 (예: DNS lookup에 의존한 보안결정)
② 에러 처리: 프로그램 입력값에 대한 검증 누락 또는 부적절한 검증, 잘못된 형식지정 (예: SQL 삽입, XQuery 삽입)
③ 시간 및 상태: 중요한 데이터를 불충분하게 캡슐화하여 인가되지 않은 사용자에게 데이터 누출 가능 (예: 잘못된 세션에 의한 정보 노출)
④ 보안 기능: 인증, 접근제어, 암호화 등의 기능을 부적절하게 구현 시 발생 (예: 부적절한 인가, 하드코드된 암호화키)
정답 ④ — ①코드 오류 설명이 맞지만 예시(DNS lookup)는 코드 오류에 속함. ②의 설명은 입력 데이터 검증 및 표현 유형. ③의 설명은 캡슐화 유형. ④만 보안 기능 = 인증·접근제어·암호화 부적절 구현이 정확함.
📄 2023 국가직 정보시스템보안 · OWASP 2017 Injection 정의★★★
2017년에 발표된 OWASP TOP 10의 항목 중 (가)에 해당되는 것은?
"(가)은(는) 신뢰할 수 없는 데이터가 명령어나 쿼리문의 일부분으로서, 인터프리터로 보내질 때 발생한다. 공격자의 악의적인 데이터는 예기치 않은 명령을 실행하거나 올바른 권한 없이 데이터에 접근하도록 인터프리터를 속일 수 있다."
① Broken Access Control
② Broken Authentication
③ Sensitive Data Exposure
④ Injection
정답 ④ — 키워드: "신뢰할 수 없는 데이터", "명령어나 쿼리문", "인터프리터로 전송", "예기치 않은 명령 실행" = Injection(인젝션). 2017 OWASP A1. SQL Injection·OS Injection·LDAP Injection 포함.
📄 2023 국가직 정보시스템보안 · CVE 형식★★★
○ CVE(Common Vulnerabilities and Exposure)는 공개적으로 알려진 소프트웨어의 보안취약점을 가리키는 고유 표기를 의미한다.
○ 취약점명은 CVE－(가)－(나) 형식으로 구성되어 있다.
① (가)일련번호 (나)소프트웨어 버전
② (가)연도 (나)소프트웨어 버전
③ (가)일련번호 (나)연도
④ (가)연도 (나)일련번호
정답 ④ — CVE 형식: CVE-[취약점 발견 연도]-[일련번호]. 예: CVE-2022-34993 → 2022년 발견, 일련번호 34993. (가)=연도, (나)=일련번호. 소프트웨어 버전은 CVE 형식에 없어요.
📄 2024 국가직 정보시스템보안 · CVSS 정의★★
보안 취약점으로 인해 실제 발생한 취약점의 등급을 매기는 공개 프레임워크는?
① CVE
② CVSS
③ CWE
④ NVD
정답 ② — CVSS(Common Vulnerability Scoring System) = 취약점 심각도를 0~10 점수로 매기는 공개 표준. CVE=취약점 고유 식별. CWE=취약점 유형 분류. NVD=미국 NIST 취약점 DB.
📄 2025 국가직 정보시스템보안 · OWASP 2021 SSRF★★★
2021년에 발표된 OWASP Top 10의 항목 중, 서버가 적절한 검증 절차를 거치지 않은 요청을 처리하여, 접근통제를 우회하는 방식으로 서버의 내부 데이터가 탈취될 수 있는 취약점은?
① XXE
② SSRF
③ Cryptographic Failures
④ Insufficient Logging & Monitoring
정답 ② — SSRF(Server-Side Request Forgery, 서버사이드 요청 위조) = 서버가 사용자가 제공한 URL을 확인하지 않고 서버 측 쿼리를 수행. VPN·방화벽·ACL에 의해 보호되는 내부에도 위조 요청 전송 가능. 2021 OWASP A10(신규). XXE는 2017/2020에 있었지만 2021에서 사라짐.
📄 2019 국가직 SW 개발보안 가이드 · 분석·설계 단계 오답★★
전자정부 SW 개발·운영자를 위한 소프트웨어 개발보안 가이드 상 분석·설계 단계 보안요구항목과 설명으로 옳지 않은 것은?
① 인증 수행 제한 — 인증 반복시도 제한 및 인증실패 등에 대한 인증제한 기능 설계
② 암호키 관리 — 암호키 생성, 분배, 접근, 파기 등 안전하게 암호키 생명주기 관리 방법 설계
③ 예외 처리 — 보안기능 동작을 위해 사용되는 입력값과 함수의 외부입력값 및 수행결과에 대한 처리방법 설계
④ 시스템 자원 접근 및 명령어 수행 입력값 검증 — 시스템 자원접근·명령어 수행에 사용되는 입력값에 대한 유효성 검증방법 설계
정답 ③ — ③의 설명은 입력 데이터 검증 단계 보안요구항목이에요. 예외 처리(Error Handling)는 에러 발생 시 불필요한 시스템 정보가 외부로 노출되지 않도록 하는 활동이에요. 설명이 뒤바뀌었어요.
