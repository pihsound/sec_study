1
Telnet의 시대 — 왜 SSH가 필요했나

1990년대 초, 시스템 관리자들은 원격에서 서버를 관리하기 위해 Telnet을 사용했어요. 원리는 단순해요 — 터미널 명령어를 네트워크로 전송하는 것. 그런데 문제가 있었어요.

Telnet은 모든 것을 평문으로 전송합니다.
비밀번호를 입력하면? 네트워크에서 p-a-s-s-w-o-r-d가 그대로 날아다녀요.
스니핑 도구 하나만 있으면 누구나 잡을 수 있어요.

1995년, 핀란드 헬싱키 대학교 연구자 타투 일로넨(Tatu Ylönen)이 자신의 대학 네트워크에서 스니핑 공격이 발생했다는 소식을 들었어요. 그는 몇 달 만에 SSH(Secure Shell)를 개발해 인터넷에 무료로 공개했어요.

SSH의 아이디어는 명확해요. Telnet처럼 원격 접속하되, 모든 통신을 암호화하자. 사용자 입장에서는 Telnet과 똑같이 쓸 수 있는데, 네트워크에서는 아무것도 읽을 수 없는 암호문만 보이는 것이죠.

💡 SSH가 해결한 3가지 문제

① 기밀성 — 모든 통신 암호화. 스니핑해도 읽을 수 없음.
② 서버 인증 — "내가 진짜 목적지 서버에 접속한 건지" 검증. 중간자 공격 방지.
③ 무결성 — 전송 중 데이터 변조 여부 탐지.

2
TCP — 신뢰성의 탄생

인터넷의 핵심 원칙은 "패킷"이에요. 데이터를 작은 조각으로 쪼개서 각자 다른 경로로 목적지까지 보내는 방식이에요. 빠르고 효율적이지만, 문제가 있어요.

패킷이 중간에 사라질 수 있어요. 순서가 뒤바뀔 수 있어요. 손상될 수 있어요. 이걸 IP 프로토콜만으로는 해결할 수 없었어요. IP는 그냥 "최선을 다해 보내드리겠습니다"라고만 할 뿐, 도착 보장을 안 해요.

그래서 TCP(Transmission Control Protocol)가 나왔어요.
"전 패킷이 빠짐없이, 순서대로, 손상 없이 도착했는지 제가 책임지겠습니다."
연결을 맺고(3-way), 데이터를 보내고, 연결을 끊는(4-way) 구조로 신뢰성을 보장해요.

오늘날 HTTP(웹), HTTPS, FTP, SSH, SMTP — 신뢰성이 필요한 거의 모든 인터넷 서비스가 TCP 위에서 동작해요. TCP는 인터넷 통신의 근간이에요.

SSH 정의 및 개요
★★★
SSH 정의

SSH(Secure Shell)는 Telnet·FTP 등을 대체하는 암호화된 원격 접속 프로토콜. TCP 기반. 포트 22번. IETF RFC 4250~4256 표준. 서버 인증·기밀성·무결성 제공. 완전 순방향 비밀성(PFS) 지원.

SSH 3가지 구성 프로토콜
★★★ 최빈출
핵심 암기: SSH = 전송층 프로토콜 + 사용자 인증 프로토콜 + 연결 프로토콜 — 3개가 TCP 위에서 계층적으로 동작
SSH 프로토콜 스택 — 3가지 구성 요소
①
SSH 전송 계층 프로토콜 (Transport Layer Protocol)
서버 인증·데이터 기밀성·데이터 무결성을 제공. 완전 순방향 비밀성(PFS) 지원 — 세션키나 영구 개인키가 손상되더라도 이전 세션의 기밀성에 영향 없음. 전송 계층에서는 옵션으로 압축 제공 가능. ★ "논리 채널 다중화"는 전송층 역할이 아님 → 연결 프로토콜 역할!
서버인증·기밀성·무결성·PFS·압축(옵션)
②
사용자 인증 프로토콜 (User Authentication Protocol)
서버에게 클라이언트 측 사용자를 인증하는 역할. 비밀번호 인증, 공개키 인증, 호스트 기반 인증 등 여러 방식 지원. ① 전송 계층 프로토콜 위에서 동작.
클라이언트 → 서버 방향 사용자 인증
③
연결 프로토콜 (Connection Protocol)
하나의 기본 SSH 연결을 사용하여 여러 개의 논리적 통신 채널을 다중화(multiplex). 예: 하나의 SSH 연결로 터미널·파일 전송·포트 포워딩을 동시에 처리. 터널링 지원.
논리 채널 다중화 · 터널링
⚠함정 — "SSH 전송계층이 논리 채널 다중화를 수행한다" (2023 지방직)
④ "SSH 전송계층 프로토콜은 다수의 기본 SSH 연결을 사용하여 통신채널을 다중화한다" → ✗ 오답

논리 채널 다중화 = 연결 프로토콜의 역할. 전송 계층 프로토콜은 서버 인증·기밀성·무결성·PFS·압축(옵션) 담당. SSH는 TCP만 사용 (UDP 아님).

① "SSH는 TCP 및 UDP 기반의 보안 프로토콜이다" → ✗ 오답 — SSH는 TCP만 사용

TCP 정의

4
3-way 핸드셰이크 — 연결 수립
★★★
TCP 3-way Handshake — 연결 수립 과정
①
SYN — 클라이언트 → 서버 (연결 요청)
클라이언트가 서버에 연결 요청. SYN=1, ACK=0. 초기 순서 번호(ISN) 포함. 클라이언트 상태: CLOSED → SYN_SENT. 서버 상태: LISTEN → SYN_RECEIVED.
SYN=1, ACK=0, Seq=x
②
SYN + ACK — 서버 → 클라이언트 (요청 수락)
서버가 클라이언트의 SYN을 받고 수락. SYN=1, ACK=1. 서버의 ISN과 클라이언트 Seq+1 (ACK 번호)를 포함. 서버 상태: SYN_RECEIVED → ESTABLISHED 준비.
SYN=1, ACK=1, Seq=y, Ack=x+1
③
ACK — 클라이언트 → 서버 (확인)
클라이언트가 서버의 SYN+ACK를 받고 확인. SYN=0, ACK=1. 양측 모두 ESTABLISHED 상태. 이후 데이터 전송 시작.
SYN=0, ACK=1, Ack=y+1
5
4-way 핸드셰이크 — 연결 해제
★★★
TCP 4-way Handshake — 연결 해제 과정 (정상 종료)
💡 3-way vs 4-way — 왜 연결 해제는 4단계인가?

연결 수립(3-way): 양측이 동시에 준비 가능 → SYN+ACK를 합쳐서 전송 가능.
연결 해제(4-way): 서버가 FIN을 받아도 데이터를 다 보내야 해요. 그래서 ACK 먼저 보내고(②), 데이터 다 보낸 후 FIN을 별도로 보내요(③). 즉 ACK와 FIN을 합칠 수 없어서 4단계가 필요.

SYN Flooding 공격
★★★
SYN Flooding 공격 정의

TCP의 3-way 핸드셰이크 구조적 취약점을 이용한 DoS 공격. 공격자가 존재하지 않는 출발지 IP로 위조된 SYN 패킷을 대량 전송 → 서버가 SYN_RECEIVED 상태로 대기하는 반연결(Half-Open)을 쌓아 연결 큐를 고갈시켜 정상 사용자 접속을 차단.

SYN Flooding 공격 vs 정상 3-way 핸드셰이크 비교
항목내용
공격 원리위조된 출발지 IP로 SYN만 대량 전송 → 서버 반연결 큐 고갈
취약점TCP 3-way 핸드셰이크 구조 — SYN_RECEIVED 상태에서 ACK 대기
대응책 ①대기 큐(SYN Queue)의 크기를 늘린다
대응책 ②최대 접속 대기 시간을 줄인다
대응책 ③SYN Cookie — 서버가 상태를 저장하지 않고 쿠키값으로 검증
대응책 ④방화벽에서 동일 IP의 SYN 패킷 임계값 초과 시 차단

FTP 정의 및 특성
★★★
FTP 정의

FTP(File Transfer Protocol)는 파일 전송을 위한 가장 기본 프로토콜. RFC 114, 1971년 제정. TCP 기반. 제어 연결 TCP 21번 / 데이터 연결 TCP 20번(능동 모드). 대화형 클라이언트-서버 통신. 로그인 인증 후 파일 전송. 텍스트·바이너리 모두 지원.

제어 포트
TCP 21번 — 명령·응답. 연결 유지.
데이터 포트
TCP 20번 — 실제 파일 데이터. 능동 모드 기준.
파일 형식

능동 모드 (Active Mode)
★★★ 최빈출
핵심: 능동 모드 = 데이터 연결을 서버가 시작 / 수동 모드 = 데이터 연결을 클라이언트가 시작
FTP 능동(Active) 모드 — 데이터 연결을 서버가 시작
①
클라이언트 → 서버 21번으로 제어 연결 수립
클라이언트가 서버 21번 포트에 제어 연결을 맺고, PORT 명령으로 자신의 데이터 포트 번호(예: 5150)를 서버에게 알린다.
클라이언트 → 서버 21번 (제어)
②
서버 20번 → 클라이언트 데이터 포트로 접속 (핵심!)
서버가 자신의 20번 포트에서 클라이언트의 데이터 포트(5150)로 접속을 시작한다. "능동(Active)"이라는 이름은 서버가 능동적으로 클라이언트에게 접속하기 때문.
서버 20번 → 클라이언트 (데이터) ← 능동 모드 핵심
③
데이터 전송
클라이언트가 ACK로 응답하면 데이터 전송 시작. 문제: 클라이언트 방화벽이 외부(서버)에서 들어오는 연결을 차단하면 작동 안 됨 → 수동 모드 필요.
2
수동 모드 (Passive Mode)
★★★

데이터 연결 시작서버가 클라이언트에게 접속클라이언트가 서버에 접속
서버 데이터 포트TCP 20번 (고정)1024번 이상 임의 포트
명령PORT 명령 (클라가 자기 포트 알림)PASV 명령 (클라가 수동 모드 요청)
방화벽 통과클라이언트 방화벽에서 차단될 수 있음방화벽 통과 용이 (클라가 아웃바운드)
서버 방화벽단순1024번 이상 포트 범위 열어야 함
⚠함정 — 능동/수동 모드에서 "누가" 데이터 연결을 시작하는지
"수동 모드에서 데이터 연결은 서버가 시작한다" → ✗ 오답

수동(Passive) = 클라이언트가 서버의 랜덤 포트로 접속. 능동(Active) = 서버의 20번 포트에서 클라이언트로 접속. 이름이 헷갈릴 수 있어요. "수동"은 서버가 수동적으로 기다린다는 뜻.

"FTP 능동 모드에서 서버 데이터 포트는 임의 포트를 사용한다" → ✗ 오답

능동 모드 서버 데이터 포트 = TCP 20번 고정. 임의 포트(1024 이상) = 수동 모드의 서버.

1
FTP 3가지 공격 유형
★★★
①
FTP 바운스 공격 (FTP Bounce Attack)
FTP 서버가 데이터를 전송할 때 목적지를 검사하지 않는 취약점을 이용. PORT 명령의 목적지 IP를 제3의 호스트로 지정하여 FTP 서버가 다른 서버를 공격하도록 만드는 공격. 공격자의 IP가 드러나지 않아 추적 어려움.
대응: FTP 서버의 20번 포트를 통한 외부 접속 제한. PORT 명령의 IP 검증.
FTP 서버 → 제3의 호스트 공격 / 추적 어려움
②
익명 FTP 공격 (Anonymous FTP Attack)
익명(Anonymous) FTP 서버의 취약점 악용. 보안 절차 없이 익명 사용자에게 FTP 서버 접근을 허용하는 경우, 쓰기 권한까지 있으면 악성 코드 업로드·배포 가능.
대응: 익명 FTP 사용 시 쓰기 권한과 실행 권한 제한. 불필요한 항목을 /etc/passwd에서 제거. /etc/ftpusers에 root 등 중요 계정 등록 금지.
익명 + 쓰기권한 = 악성코드 배포 위험
③
TFTP 공격
TFTP(Trivial FTP)의 구조적 취약점 악용. TCP 대신 UDP 사용. 인증 절차 없음. 설정이 잘못될 경우 누구나 호스트에 접근하여 파일 유출 가능. 클라이언트 설치 없이 접근 가능.
대응: 불필요한 TFTP 서비스 비활성화. 필요시 Secret Mode로 운영. 접근 IP 필터링.
UDP 69번 / 인증없음 / 누구나 접근
공격 유형원리대응책
FTP 바운스PORT 명령 목적지를 제3 호스트로 지정 → FTP 서버가 대신 공격20번 포트 통한 외부접속 제한, PORT IP 검증
익명 FTP인증 없는 익명 접근 + 쓰기 권한 → 악성코드 업로드쓰기·실행 권한 제한, /etc/ftpusers에 root 등록 금지
TFTPUDP·인증없음 구조 → 누구나 파일 접근서비스 비활성화, Secret Mode 운영
1
FTP 보안 강화 방법
★★★
FTPS
FTP over SSL/TLS. SSL/TLS로 암호화. 제어·데이터 채널 모두 암호화. 포트: 명시적=990, 암시적=21. 기존 FTP 구조 유지하면서 암호화 추가.
SFTP
SSH File Transfer Protocol. FTP와 무관. SSH(TCP 22번) 위에서 동작. 포트가 22번 하나뿐. 단일 채널. 방화벽 통과 쉬움. 현재 가장 권장.
SCP
Secure Copy Protocol. SSH 기반 파일 복사. TCP 22번. 단방향 파일 전송에 특화. SFTP보다 단순.
서비스 금지
FTP가 반드시 필요하지 않으면 서비스 자체를 비활성화. 가장 근본적인 보안 대책.
보안 프로토콜기반포트특징
FTPSFTP + SSL/TLS21 또는 990기존 FTP 구조 유지. 인증서 필요.
SFTPSSH 프로토콜22FTP와 다른 프로토콜. 단일 채널. 방화벽 통과 쉬움. 현재 권장.
SCPSSH 프로토콜22SSH 기반 파일 복사. 단순하고 빠름.
⚠함정 — "SFTP는 FTP에 보안을 추가한 것이다" / "FTPS와 SFTP는 같다"
"SFTP는 FTP에 SSH 암호화를 추가한 프로토콜이다" → ✗ 오답

SFTP = SSH 기반의 완전히 별도 프로토콜. FTP와 무관. 포트도 22번(SSH). FTPS가 FTP + SSL/TLS 암호화를 추가한 것. 이름이 비슷해 혼동 주의.

"익명 FTP는 모든 사용자에게 동일한 아이디와 이메일 주소를 패스워드로 요구한다" → ✓ 정답

익명(Anonymous) FTP의 특성 — ID로 "anonymous"를 사용하고, 비밀번호 자리에 이메일 주소를 요구하는 방식이 일반적.

② Smurf 공격은 출발지와 목적지의 IP 주소를 공격대상의 IP 주소로 변조하여 전송하며, 자신에게 응답하는 과정을 반복하게 하여 과부하를 유발한다.

④ Ping of Death 공격은 아주 작은 조각의 ICMP Ping 패킷을 처리하게 하여 공격 대상 시스템의 성능저하 및 마비를 일으키는 공격이다.
정답 ② — Smurf 공격: 출발지 IP를 피해자 IP로 위조한 ICMP Echo를 브로드캐스트 주소로 전송 → 네트워크 내 모든 호스트가 피해자에게 ICMP Reply 전송 → 과부하. "자신에게 응답 반복" = Land 공격 설명.

④ SSH는 파일 전송 기능(SCP/SFTP)을 제공한다.
정답 ③ — SSH는 TCP만 사용. UDP 기반이 아님. DTLS처럼 UDP 기반 보안 프로토콜과 혼동 주의. SSH: TCP 22번, Telnet: TCP 23번.
📄 복합 — TFTP 특성 (UDP + 인증 없음)★★
TCP 대신에 UDP 69번 포트를 사용하고, 사용자 인증 절차를 요구하지 않기 때문에 누구나 호스트로부터 파일을 가져갈 수 있도록 고안된 프로토콜은?
① HTTP
② FTP
③ TFTP
④ SOAP
정답 ③ — TFTP(Trivial FTP): UDP 69번, 인증 없음, 단순 파일 전송. FTP는 TCP 20/21번. HTTP는 TCP 80번. TFTP는 라우터 부팅 시 설정 파일 로드 등에 사용.

SSH 전송층 ≠논리 채널 다중화는 연결 프로토콜 역할. 전송층 아님!

📄 2016 정보시스템보안 · FTP 보안 설명 오답★★★
FTP 보안에 대한 설명으로 옳지 않은 것은?
① 임의의 계정으로 로그인 시도를 반복하여 패스워드를 유추할 수 있는 취약점이 있다.

③ 익명 FTP는 모든 사용자에게 동일한 사용자 아이디와 유효한 사용자 이메일 주소를 패스워드로 요구한다.
④ FTP 보안 대책으로 반드시 서비스 사용이 필요하지 않은 경우 FTP 서비스 사용을 금지하는 것이 좋다.
정답 ③ — 익명 FTP는 ID로 "anonymous"를 사용하되, 비밀번호에는 유효한 이메일 주소를 요구하지 않고 형식만 맞으면 됨. 실제로 아무 이메일 형식이나 입력해도 접속 가능한 경우가 많음. ④는 올바른 대책.

📄 2020 경찰간부 · FTP 보안 대책 옳고 그름★★
FTP 보안 대책으로 옳은 것을 모두 고른 것은?
가. Bounce Attack 예방을 위해 20번 포트를 통한 명령은 제한한다.
나. TFTP가 필요한 경우 Secret Mode로 운영한다.
다. /etc/passwd에서 Anonymous FTP에 불필요한 항목을 제거한다.
라. /etc/ftpusers에 등록된 사용자만이 FTP에 접근할 수 있으므로, root와 같은 중요 계정은 등록하지 않는다.
① 가, 다
② 나, 라
③ 나, 다
④ 다, 라
정답 ④ — 가는 틀림: Bounce Attack 대응은 20번 포트 외부 접속 제한(명령 제한이 아님). 나는 틀림: TFTP Secret Mode는 일부 구현에서만 있는 옵션. 다는 옳음: 불필요 항목 제거. 라는 옳음: /etc/ftpusers의 계정은 FTP 접근 금지이므로 root는 반드시 등록해야 함.
📄 복합 · FTP 포트 번호 구분★★★
FTP에서 데이터 전송에 사용되는 포트 번호는?
① TCP 21번
② TCP 20번
③ UDP 69번
④ TCP 22번
정답 ② — TCP 20번 = FTP 데이터 / TCP 21번 = FTP 제어(명령) / UDP 69번 = TFTP / TCP 22번 = SSH.
📄 복합 · FTP 능동/수동 모드 비교★★★
FTP 수동(Passive) 모드에 대한 설명으로 옳은 것은?
① 서버가 클라이언트의 데이터 포트로 능동적으로 접속한다.
② 서버 데이터 채널은 항상 20번 포트를 사용한다.
③ 클라이언트가 서버의 1024번 이상 임의 포트로 데이터 연결을 시작한다.
④ 방화벽 환경에서 클라이언트 인바운드 포트를 열어야 한다.
정답 ③ — ①③은 반대(능동=서버→클라, 수동=클라→서버). ②20번 포트 고정=능동 모드. ④수동 모드는 클라이언트 아웃바운드 연결 → 방화벽 통과 용이. 인바운드 열 필요 없음.
📄 복합 · FTP 바운스 공격★★
FTP Bounce 공격에 대한 설명으로 옳은 것은?
① TFTP의 취약점을 이용하여 파일을 불법으로 복제하는 공격이다.
② 인증 없이 FTP 서버에 접근하여 악성 코드를 업로드하는 공격이다.
③ FTP 서버가 데이터 전송 시 목적지를 검사하지 않는 취약점을 이용해 제3의 호스트를 공격하는 방식이다.
④ FTP 로그인 시 패스워드를 반복 시도하여 유추하는 공격이다.
정답 ③ — FTP Bounce = PORT 명령 목적지를 제3의 호스트로 지정 → FTP 서버가 대신 공격. 공격자 IP 추적 어려움이 핵심. ①=TFTP 공격, ②=익명FTP 공격, ④=브루트포스.
📄 복합 · SFTP vs FTPS 구분★★
SFTP에 대한 설명으로 옳은 것은?
① FTP에 SSL/TLS 암호화를 추가한 프로토콜이다.
② 기본 포트로 TCP 21번을 사용한다.
③ 제어 채널과 데이터 채널을 분리하여 운용한다.
④ SSH 프로토콜을 기반으로 하며 TCP 22번 포트를 사용한다.
정답 ④ — SFTP = SSH 기반, TCP 22번, 단일 채널. ①은 FTPS 설명. ②21번은 FTP/FTPS. ③이중 채널은 FTP/FTPS의 특성.
💡 FTP 핵심 암기
FTP 포트제어 = TCP 21번 / 데이터 = TCP 20번 (능동 모드)
TFTP 포트UDP 69번 / 인증 없음 / 누구나 접근 가능
능동 모드서버의 20번 → 클라이언트 데이터 포트 접속. PORT 명령.
