1
웹이 열리던 날, 보안의 구멍도 함께 열렸다

1990년대 중반, 웹(WWW)이 폭발적으로 보급되면서 기업들은 앞다투어 홈페이지를 만들었어요. 초기 웹 개발자들의 관심은 오직 "어떻게 하면 더 많은 정보를 빠르게 보여줄 수 있을까"였어요. 보안은 나중 문제였죠. 이 틈에서 공격자들은 웹의 구조적 특성을 악용하는 방법을 하나씩 발견하기 시작했어요.

웹의 근본적인 취약점은 단순해요. 사용자가 입력한 데이터를 서버가 신뢰한다는 점이에요. 웹 폼에 이름을 입력하면 서버는 그 값이 정말 이름인지, 아니면 악성 코드인지 검증하지 않고 처리하는 경우가 많아요. 이 하나의 설계 실수에서 XSS, SQL Injection, CSRF 등 수많은 공격이 탄생했어요.

1995년, 처음으로 웹 스크립트 삽입 공격이 보고됐어요. 공격자가 게시판에 악성 스크립트를 심고, 다른 사용자가 그 글을 읽으면 스크립트가 실행된다는 사실이 발견됐어요. 이게 XSS(Cross-Site Scripting)의 시작이에요. 2001년에는 SQL 쿼리 조작으로 DB 인증을 우회하는 SQL Injection이 널리 알려졌고, 2001년 CSRF도 학계에 처음 보고됐어요.

이 공격들이 특히 위험한 이유가 있어요. 방화벽이나 IDS가 막아주지 못해요. 왜냐하면 공격 트래픽이 합법적인 HTTP 80번 포트를 통해 들어오거든요. 네트워크 관점에서는 정상 웹 요청이에요. 이 때문에 웹 전용 보안 장비인 WAF(Web Application Firewall)가 필요해졌어요.

웹 보안 취약점의 공통 원인:
"사용자 입력값을 검증(Validation)·이스케이프(Escape)하지 않고 그대로 사용한다."

XSS → 입력값을 HTML에 그대로 출력 → 스크립트 실행
SQL Injection → 입력값을 SQL 쿼리에 그대로 삽입 → DB 조작
경로 조작 → 입력값을 파일 경로에 그대로 사용 → 시스템 파일 접근
2
공격자의 침투 경로 — 웹쉘과 리버스 텔넷

공격자가 서버 권한을 탈취하려면 단계가 있어요. 취약점 발견 → 파일 업로드 → 웹쉘 설치 → 리버스 텔넷으로 원격 제어가 일반적인 공격 경로예요. 각 단계는 앞 단계의 성공을 전제로 해요.

파일 업로드 취약점은 문이에요. 확장자나 내용 검증 없이 파일을 업로드할 수 있다면, 공격자는 PHP·JSP·ASP 같은 서버 사이드 스크립트 파일을 올릴 수 있어요. 이게 웹쉘(Web Shell)이에요. 웹쉘은 HTTP를 통해 서버 OS 명령어를 실행할 수 있게 해주는 악성 스크립트예요. 브라우저에서 웹쉘 URL을 접근하면 서버에서 임의의 명령이 실행돼요.

그런데 문제가 있어요. 방화벽이 외부에서 서버로 들어오는(인바운드) 연결을 차단하면, 공격자가 서버에 직접 접속하기 어려워요. 이때 쓰는 게 리버스 텔넷(Reverse Telnet)이에요. 서버가 먼저 공격자 PC로 연결을 시도해요. 방화벽 입장에서는 서버에서 나가는(아웃바운드) 연결이니까 차단하지 않아요. 80번(HTTP) 포트를 이용하면 더욱 탐지하기 어려워요.

이 흐름을 이해하면 대응책도 자연스럽게 이해돼요. 파일 업로드 제한(확장자 화이트리스트) → 웹쉘 차단. 아웃바운드 트래픽도 차단 → 리버스 텔넷 차단. 업로드 디렉터리 실행 권한 제거 → 웹쉘 실행 차단. 모든 대응이 하나로 연결되는 구조예요.

1
XSS — Cross-Site Scripting
★★★ 최빈출
XSS 정의

XSS(Cross-Site Scripting)는 공격자가 작성한 악성 스크립트가 다른 사용자의 브라우저에서 실행되는 취약점. 웹 브라우저에서 사용자가 입력하는 값에 대한 유효성을 검증하지 않고 그대로 웹 서버에 전송할 때 발생. 공격자의 의도적인 악성 스크립트를 웹 서버에 저장하는 것이 1차 목적. 다른 사용자 브라우저에서 적절한 검증 없이 실행되므로 세션을 탈취하거나, 웹 사이트를 변조하거나, 악의적인 사이트로 이동시킬 수 있음.

XSS의 핵심은 "공격자가 쓴 코드가 피해자의 브라우저에서 실행된다"는 점이에요. 공격자가 직접 피해자를 공격하는 게 아니라, 취약한 웹 서버를 매개체로 사용해요. 자바스크립트가 브라우저에서 실행된다는 특성을 악용하기 때문에, 서버 측이 아닌 클라이언트 측 공격이에요. 하지만 공격 코드를 심는 행위는 서버(게시판 등)를 통해 이뤄져요.

XSS로 무엇을 할 수 있을까요? 가장 흔한 목적은 쿠키·세션 탈취예요. 브라우저에 저장된 세션 쿠키를 자바스크립트로 읽어 공격자 서버에 전송하면, 공격자가 피해자로 로그인할 수 있어요. 또한 피싱 페이지로 이동 유도, 키로거(키 입력 가로채기), 악성코드 다운로드 등에도 활용돼요.

XSS 3가지 유형 — 공격 흐름 비교

XSS는 공격 방식에 따라 세 가지로 나뉘어요. Stored XSS(저장형)는 가장 위험한 유형이에요. 게시판·댓글 등에 악성 스크립트를 저장해 두면, 그 페이지를 읽는 모든 사용자에게 계속 공격이 가해져요. 1번 올리면 무한히 작동해요. Reflected XSS(반사형)는 URL 파라미터에 스크립트를 삽입해 피해자에게 클릭하도록 유도하는 방식이에요. 서버에 저장되지 않아요. DOM 기반 XSS는 서버와 관계없이 클라이언트 브라우저의 DOM을 조작해 실행돼요.

XSS 방어의 핵심은 입출력 값 필터링·이스케이프예요. <를 &lt;로, >를 &gt;로 변환하면 HTML 태그로 해석되지 않아요. 서버 측 검증 + 클라이언트 측 검증 모두 필요해요. Content Security Policy(CSP) 헤더 설정도 효과적이에요.

XSS 유형저장 여부공격 방식특징
Stored XSS✅ DB 영구 저장게시판·댓글에 삽입 후 열람 유도가장 위험. 1회 삽입으로 다수 피해.
Reflected XSS❌ 미저장URL 파라미터에 삽입 → 링크 전송피해자가 직접 링크 클릭 필요.
DOM 기반 XSS❌ 미저장DOM API 조작(클라이언트 측)서버 응답 없이 실행. 탐지 어려움.
⚠함정 — XSS 유형명 혼동 (2017 국가직 정보시스템보안)
"Cross Site Scripting(XSS) 공격 유형: ① Reflash XSS 공격 ② Reflected XSS 공격 ③ DOM 기반 XSS 공격 ④ Stored XSS 공격 — 해당하지 않는 것은?" → 정답: ① Reflash XSS

"Reflash"는 존재하지 않는 명칭이에요. 올바른 유형명은 Reflected(반사형)이에요. "Flash"와 "Reflected"를 헷갈리도록 만든 함정. XSS 3종 = Stored / Reflected / DOM 기반만 기억하면 돼요.

2
CSRF — Cross-Site Request Forgery
★★★
CSRF 정의

CSRF(Cross-Site Request Forgery)는 로그인한 사용자의 세션을 이용해 공격자가 의도한 행위(수정·삭제·등록·송금)를 특정 웹사이트에 요청하게 만드는 공격. 불특정 다수를 대상으로 하며, 사용자는 자신의 의지와는 무관하게 요청을 보내게 됨. 특정 웹사이트가 사용자의 웹브라우저를 신뢰하는 상태를 악용.

CSRF를 이해하려면 쿠키 기반 세션의 동작 방식을 알아야 해요. 사용자가 은행 사이트에 로그인하면 브라우저에 세션 쿠키가 저장돼요. 이후 해당 은행 사이트에 요청을 보낼 때마다 브라우저는 자동으로 쿠키를 함께 보내요. 공격자는 이 자동 전송 메커니즘을 악용해요.

공격 시나리오를 살펴볼게요. 피해자가 은행 사이트(bank.com)에 로그인한 상태에서, 공격자가 만든 악성 페이지(evil.com)를 열었어요. 그 악성 페이지에는 <img src="https://bank.com/transfer?to=attacker&amount=1000000">가 숨어 있어요. 브라우저는 이 img 태그를 처리하면서 bank.com에 자동으로 요청을 보내고, 이때 bank.com 세션 쿠키도 자동으로 함께 전송돼요. 은행 서버는 정상 요청으로 판단하고 송금을 처리해 버려요.

CSRF는 XSS와 매우 유사해 보이지만 방향이 정반대예요. XSS는 악성 스크립트가 클라이언트(브라우저)에서 실행되는 공격이고, CSRF는 악성 요청이 서버로 전송되는 공격이에요. XSS는 사용자를 공격하고, CSRF는 서버(웹사이트)를 공격해요.

CSRF 방어의 핵심은 CSRF 토큰(Anti-CSRF Token)이에요. 서버가 세션마다 고유한 토큰을 발급하고, 중요한 요청에서 이 토큰 값이 올바른지 검증해요. 공격자는 피해자의 세션 쿠키는 자동으로 얻을 수 있지만, 토큰 값은 알 수 없어요. 추가적으로 중요한 요청에는 사용자 재인증(패스워드 재확인)을 요구하는 방법도 있어요.

XSS (사이트 간 스크립팅)
피해 대상사용자(클라이언트) — 사용자 브라우저가 피해
실행 위치클라이언트 측 (사용자 브라우저)
공격 방법악성 스크립트를 웹 서버에 삽입
목적쿠키 탈취, 페이지 변조, 악성코드 배포
서버 관점서버는 피해자 역할이 아님 (매개체)
CSRF (사이트 간 요청 위조)
피해 대상서버(웹사이트) — 서버가 위조 요청 처리
실행 위치서버 측 (서버가 요청을 처리)
공격 방법로그인된 사용자의 세션을 이용한 위조 요청
목적계정 정보 변경, 송금, 게시물 등록·삭제
서버 관점서버는 정상 요청으로 오인하여 처리
⚠함정 ★★★ — CSRF와 세션 하이재킹 혼동 (2020 국가직 정보시스템보안)
"CSRF는 서버와 클라이언트에 각각 잘못된 시퀀스 넘버를 사용하여 연결된 세션에 혼란을 줌으로써 공격자가 서버와의 연결을 획득하는 방법이다" → ✗ 오답

이 설명은 세션 하이재킹(Session Hijacking)이에요. TCP 시퀀스 번호를 이용한 세션 탈취 공격이죠. CSRF는 시퀀스 번호와 전혀 무관해요. CSRF = 로그인된 사용자를 이용해 위조 요청을 서버에 전송하는 공격.

"CSRF와 XSS의 차이 — XSS 공격에서는 악성 스크립트가 클라이언트에서 실행되는 반면, CSRF 공격은 악성 스크립트를 서버에 요청한다는 차이가 있다" → ✓ 정답

SQL Injection 완전 정리
★★★ 최빈출
SQL Injection 정의

SQL Injection은 공격자가 입력값을 조작하여 원하는 SQL 구문을 실행시키는 공격. 웹에서 사용자가 입력하는 값이 DB 질의어(SQL)와 연동될 때, 입력값을 적절히 검증·처리하지 않으면 발생. 공격자는 명령어나 질의어에 특정 코드를 삽입하여 DB 인증을 우회하거나 데이터를 조작·삭제할 수 있음. OWASP TOP 10에서 2004·2007·2010·2013·2017년에 지속 포함된 핵심 취약점.

SQL Injection의 동작 원리를 코드로 이해해볼게요. 일반적인 로그인 쿼리를 보면, 사용자가 입력한 ID와 PW를 그대로 SQL 문자열에 연결해요. 여기서 PW 필드에 ' OR 1=1--을 입력하면 어떻게 될까요?

-- 정상 쿼리
SELECT * FROM 학생 WHERE ID = 'admin' AND PW = '비밀번호';

-- 공격자가 PW 입력란에 ' OR 1=1-- 를 입력하면
SELECT * FROM 학생 WHERE ID = 'admin' AND PW = '' OR 1=1--';

-- 실제로 실행되는 쿼리 (--는 주석, OR 1=1은 항상 참)
SELECT * FROM 학생 WHERE ID = 'admin' AND PW = '' OR 1=1-- (나머지 주석 처리)

쿼리 해석: AND PW = ''는 거짓이지만, OR 1=1이 추가되면서 전체 WHERE 조건이 항상 참(True)이 돼요. 결과적으로 PW가 틀려도 첫 번째 레코드(보통 관리자 계정)로 로그인이 성공해요. 이것이 SQL Injection의 가장 기본적인 형태인 인증 우회 공격이에요.

SQL Injection의 피해 범위는 인증 우회에 그치지 않아요. 데이터 전체를 조회(UNION SELECT 구문)하거나, 데이터를 삭제(DROP TABLE)하거나, 파일을 읽거나 쓸 수도 있어요. 데이터베이스 서버 권한에 따라 OS 명령어 실행까지 가능해요.

핵심 대응책은 두 가지예요. 첫째, 매개변수화 쿼리(Parameterized Query)/준비된 구문(Prepared Statement)을 사용하는 거예요. SELECT * FROM 학생 WHERE ID = ? AND PW = ?처럼 입력값 자리에 ?를 쓰면, 드라이버가 입력값을 코드가 아닌 데이터로 처리해요. 둘째, 특수문자 필터링이에요. ', ", ;, --, + 등 SQL에서 특수한 의미를 갖는 문자를 필터링해요.

⚠함정 ★★★ — SQL Injection 대응 방법 (2018 국회직)
"SQL injection: 웹에서 사용자가 입력하는 값이 DB 질의어와 연동이 되는 경우에는 클라이언트 측에서만 자바스크립트 등을 통해 사용자의 입력값을 검증하는 것으로 해결된다" → ✗ 오답

클라이언트 측(브라우저) 검증만으로는 절대 해결되지 않아요. 공격자는 브라우저 없이 직접 HTTP 요청을 조작할 수 있거든요. 서버 측 검증이 반드시 필요해요. 클라이언트 검증은 UX 편의용, 서버 검증이 실제 보안.

"SQL Injection: 전송되는 패킷을 가로채어 자신이 송신자인 것처럼 패킷을 변경하여 다시 보내는 기법이다" → ✗ 오답

이 설명은 패킷 스푸핑·세션 하이재킹의 설명이에요. SQL Injection은 패킷 가로채기와 무관하며, 입력값 조작으로 SQL 쿼리를 변형시키는 공격이에요.

1
인증(Authentication) 기능 취약점
★★
인증 기능 취약점 정의

인증(Authentication)이란 현재 요청을 보낸 사람이 실제로 주장하는 사람인지를 확인하는 과정. 적절한 인증 없이 중요 기능이 노출되는 경우 보안 취약점이 됨. 취약한 인증은 웹 어플리케이션 공통 패스워드 문제 등에서 가장 기본적인 취약점이며, 관리자 변경 기능이 별도로 만들어져 있지 않고 기본 계정·패스워드가 그대로 사용되는 경우가 많음.

인증 취약점의 가장 흔한 형태는 기본 계정 미변경이에요. 웹 서버·DB·네트워크 장비를 설치하면 기본 관리자 계정(admin/admin, root/root 등)이 설정되어 있어요. 관리자가 이를 변경하지 않으면 누구나 기본값으로 로그인할 수 있어요. 개발자도 개발 환경에서는 보안보다 편의를 우선하기 때문에, 운영 환경에서도 admin/admin 같은 취약한 계정이 남아있는 경우가 많아요.

또 다른 형태는 세션 관리 취약점이에요. 로그아웃 후에도 세션이 서버에 남아있거나, 세션 토큰이 예측 가능한 패턴(ex: 사용자 ID 기반)으로 생성되거나, HTTPS 없이 세션 쿠키가 전송되면 취약해요. 2022 국가직 기출에서 "인증 후 정상적으로 세션이 발행된 페이지의 정보를 취득하고 일정 시간 후에 재전송했을 때 정상 처리가 되는지 확인하는 방법 → 불충분한 세션 관리 취약점 점검"으로 출제됐어요.

2
부적절한 인가(Authorization) 취약점
★★
부적절한 인가 정의

인가(Authorization)란 인증된 사용자가 어떤 자원에 접근할 수 있는지 결정하는 과정. 인증에 성공한 사용자가 어떤 기능을 요청할 때, 해당 기능을 수행할 권한이 있는지 서버가 검증하지 않으면 보안 취약점이 됨. 권한을 확인하거나 변경하는 과정에서 발생.

인가 취약점의 대표 사례는 URL 직접 접근이에요. 일반 사용자가 관리자 페이지 URL(/admin/users.php)을 직접 브라우저에 입력하면 접근이 되어버리는 경우예요. 화면에서 링크를 숨겼어도 URL을 알면 접근할 수 있어요. 서버 측에서 반드시 권한을 재검증해야 해요.

또 다른 사례는 파라미터 조작이에요. 내 정보 조회 URL이 /user?id=100인데, id=101로 바꿔서 다른 사람의 정보를 볼 수 있는 경우예요. 이를 IDOR(Insecure Direct Object Reference, 불안전한 직접 객체 참조)라고도 해요. 숫자 하나만 바꾸면 다른 사용자의 데이터에 접근할 수 있어요.

3
경로 조작(Path Traversal)
★★★
경로 조작 정의

경로 조작(Path Traversal)은 사용자 입력에 대한 검증을 하지 않고 그대로 파일 경로로 사용할 때, 서버 시스템의 임의 자원에 접근하는 보안 취약점. 부모 디렉터리를 의미하는 ../를 이용하여 허가된 디렉터리를 벗어나 시스템 중요 파일을 읽거나 삭제할 수 있음.

경로 조작의 원리는 단순해요. 파일 다운로드 기능이 /download?file=report.pdf처럼 파일명을 파라미터로 받는다면, 공격자는 이를 /download?file=../../../etc/passwd로 바꿔요. ../는 "한 단계 위 디렉터리로 이동"을 의미하므로, 웹 서비스 디렉터리를 벗어나 시스템 파일에 접근할 수 있어요.

실제 공격에선 URL 인코딩(%2e%2e%2f)이나 이중 인코딩(%252e%252e%252f)으로 필터링을 우회하려는 시도도 많아요. 그래서 단순히 ../ 문자열만 필터링하는 방식은 불충분하고, 경로를 정규화(Canonical Path)한 후 허용된 디렉터리 내에 있는지 검증해야 해요.

2022년 국가직 정보시스템보안에서는 "../를 이용해서 임의의 경로가 포함된 값으로 웹페이지 파라미터를 변조한 후 해당 경로의 파일 내용이 표시되는지 확인하는 방법이 점검하는 취약점은?" → 경로 추적(Path Traversal)으로 출제됐어요. 디렉터리 인덱싱과 혼동하지 않도록 주의해야 해요. 디렉터리 인덱싱은 특정 URL 경로에 직접 접근했을 때 디렉터리 파일 목록이 보이는 취약점이에요.

1
파일 업로드 제한 부재
★★★
파일 업로드 취약점 정의

파일 업로드 취약점은 업로드되는 파일의 유효성을 검증하지 않아 실행 가능한 스크립트 파일(asp, php, jsp 등)이 업로드될 수 있는 보안 취약점. 공격자는 악성 스크립트(웹쉘)를 업로드하고 이를 실행하여 시스템 명령을 수행하거나 내부 침투 기회를 얻음. 리버스 텔넷 같은 원격 제어를 얻기 위해 반드시 거쳐야 하는 공격 단계.

파일 업로드 취약점이 위험한 이유는 단순히 악성 파일이 서버에 올라가는 게 아니라, 그 파일이 서버에서 실행될 수 있다는 점이에요. PHP 파일을 업로드하고 웹 브라우저로 그 파일의 URL에 접근하면, 웹 서버가 해당 PHP 코드를 실행해요. 이렇게 만들어지는 게 웹쉘이에요.

업로드 취약점이 발생하는 이유는 검증의 부재 또는 우회 가능한 검증 방식 때문이에요. 단순히 파일 확장자만 확인하는 경우, 공격자는 shell.php를 shell.php.jpg로 바꿔 우회를 시도해요. 또는 Content-Type 헤더를 image/jpeg로 변조해서 우회하기도 해요. 그래서 확장자 확인, MIME 타입 확인, 파일 내용(Magic Byte) 확인 모두 필요해요.

확장자 화이트리스트
허용할 확장자(.jpg·.png·.pdf)만 명시하고 나머지 차단. 블랙리스트 방식은 우회 가능.
업로드 경로 격리
업로드 디렉터리를 web document root 밖에 저장. 직접 URL 접근 불가.
실행 권한 제거
업로드 디렉터리에 스크립트 실행 권한 제거. 업로드돼도 실행 불가.
파일명 변경
저장 시 임의 파일명으로 변경. 업로드 파일명에 필터링 설정.
2
웹쉘 (Web Shell)
★★★
웹쉘 정의

웹쉘(Web Shell)은 클라이언트에서 웹 서버에 명령을 전달하기 위한 목적으로 만든 스크립트 프로그램. 서버 사이드 스크립트(ASP·PHP·JSP 등)로 작성되어 업로드됨. 설치되면 기존 서비스를 계속 작동시키면서 공격자가 만든 명령을 실행하게 됨. HTTP를 통해 동작하기 때문에 서버 측에서 실행되는 원격 접속 도구.

웹쉘의 핵심 특성은 HTTP를 통해 OS 명령어를 실행한다는 점이에요. 일반적인 텔넷이나 SSH 접속은 해당 포트가 열려있어야 하고 방화벽에 차단될 수 있어요. 하지만 웹쉘은 이미 열려있는 80번(HTTP) 포트를 사용해요. 방화벽이 있어도 통과할 수 있어요.

간단한 웹쉘의 예시를 보면, PHP로 한 줄만 써도 돼요. <?php system($_GET['cmd']); ?>. 이 파일을 서버에 업로드하고 http://target.com/uploads/shell.php?cmd=ls를 접근하면, 서버에서 ls 명령이 실행되고 결과가 브라우저에 표시돼요. 이것 하나로 파일 목록 확인, 파일 내용 출력, 시스템 정보 수집 등이 가능해요.

웹쉘은 탐지하기 어렵고 비교적 쉽게 제어할 수 있어 흔히 쓰이는 공격 방법이에요. HTTP 트래픽처럼 보이기 때문에 네트워크 수준에서 탐지가 어렵고, 정상적인 웹 요청과 외형이 같아요. 대응으로는 파일 업로드 취약점 제거(근본 원인 제거), 업로드 파일 실행 권한 제거, 무결성 검사 도구(Tripwire 등)로 파일 변경 탐지, 웹 방화벽(WAF) 시그니처 적용 등이 있어요.

⚠함정 — 웹쉘 대응 방법 (2024 국가직 정보시스템보안)
웹 셸(web shell) 공격에 대한 대응 방법으로 옳지 않은 것은? — "업로드 파일에 대한 실행 금지 / 업로드 파일 위치 격리 / 업로드 가능한 파일 형식 제한 / ④ 업로드 파일명에 대한 필터링 설정" → 모두 옳음. 이 문제에서 오답인 것은 다른 선지.

웹쉘 대응 4원칙: ①실행 금지 ②위치 격리 ③형식 제한 ④파일명 필터링. 모두 유효한 대응. 서버에 업로드된 파일이 직접 실행되지 않도록 다중 방어 적용 필요.

3
리버스 텔넷 (Reverse Telnet)
★★★
리버스 텔넷 정의

리버스 텔넷(Reverse Telnet)은 공격 대상 서버가 먼저 공격자 PC로 연결을 시도하게 하여 방화벽의 인바운드 차단 정책을 우회하는 공격 기법. 웹 서버가 80번 포트(HTTP)를 통해 나가는 연결을 이용하면 방화벽이 차단하지 않아 탐지하기 어려움. 파일 업로드 취약점으로 설치한 웹쉘을 통해 실행됨.

일반적인 원격 접속 방향은 공격자 → 서버예요. 공격자가 텔넷·SSH로 서버에 접속하려면 해당 포트(23·22번)가 열려있어야 해요. 방화벽이 이 포트로의 인바운드 연결을 차단하면 공격자는 접속할 수 없어요.

리버스 텔넷은 방향을 바꿔요. 서버 → 공격자로 연결해요. 공격자는 자신의 PC에서 특정 포트를 열고 기다려요. 웹쉘을 통해 서버에서 공격자 PC로 접속 명령을 실행해요. 방화벽 입장에서는 서버에서 나가는(아웃바운드) 연결이라 차단하지 않아요. 특히 80번(HTTP) 포트를 이용하면 정상 웹 통신처럼 보여서 더 탐지하기 어려워요.

리버스 텔넷 대응책은 파일 업로드 취약점을 먼저 막아야 해요. 웹쉘이 없으면 리버스 텔넷 명령을 실행할 방법이 없기 때문이에요. 추가로 서버에서 외부로 나가는 아웃바운드 트래픽도 불필요한 연결을 차단하는 게 좋아요. 많은 기업이 인바운드만 차단하고 아웃바운드는 허용하는데, 이 정책의 빈틈을 리버스 텔넷이 파고드는 거예요.

정답 ④ — "명령어·질의어에 코드 삽입 → DB 인증 우회·데이터 조작" = SQL Injection. 직접 객체 참조(IDOR)는 권한 없이 다른 사용자 데이터 접근. CSRF는 로그인 세션을 이용한 위조 요청.

① 명령 삽입 취약점

📄 2017 국가직 정보시스템보안 · XSS 유형 오답 찾기★★★
Cross Site Scripting(XSS) 공격유형에 해당하지 않는 것은?
① Reflash XSS 공격
② Reflected XSS 공격
③ DOM(Document Object Model) 기반 XSS 공격
④ Stored XSS 공격
정답 ① — "Reflash"는 존재하지 않는 공격 유형. XSS 3종 = Stored(저장형) / Reflected(반사형) / DOM 기반. "Reflected"와 비슷하게 만든 함정. Flash는 Adobe Flash로 혼동 유도.
📄 2018 국회직 9급 · 웹 공격 유형 종합★★★
웹 공격의 유형에 대한 설명으로 옳지 않은 것은?
① XSS: 저장·반사·DOM 기반 XSS로 분류. 입출력값 검증·인코딩으로 대응.
② SQL injection: 클라이언트 측에서만 자바스크립트 등을 통해 사용자의 입력값을 검증하는 것으로 해결된다.
③ CSRF: 사용자가 자신의 의도와는 무관하게 공격자가 의도한 행위를 특정 웹사이트에 요청하게 만드는 공격.
④ 쿠키획득 공격: 로그인된 사용자의 쿠키값을 XSS로 획득하여 로그인 가능.
정답 ② — 클라이언트 측 검증만으로는 SQL Injection 방어 불가. 공격자는 브라우저 우회 후 직접 HTTP 요청 조작 가능. 반드시 서버 측 검증(Prepared Statement·특수문자 필터링)이 필요.
📄 2019 경찰공무원 · Reflected XSS vs CSRF★★★
㉠ 공격: URL에 악성 스크립트를 포함시킨 메일을 보내고 사용자가 클릭하면 서버로부터 반송된 오류 메시지에 포함된 악성 스크립트가 실행되는 방법.
㉡ 공격: 로그인 상태인 사용자의 세션 쿠키와 다른 인증정보를 취약한 웹 앱에 자동으로 포함시키고, 공격자가 의도한 HTTP 요청을 사용자가 보낸 것처럼 자동으로 보내는 방법.
① ㉠ Reflected XSS, ㉡ CSRF
② ㉠ SQL 인젝션, ㉡ Reflected XSS
③ ㉠ CSRF, ㉡ SQL 인젝션
④ ㉠ Reflected XSS, ㉡ S-HTTP
정답 ① — ㉠: URL 파라미터에 스크립트 삽입 + 클릭 유도 + 서버가 반사 = Reflected XSS. ㉡: 로그인 세션 + 쿠키 자동 포함 + 위조 HTTP 요청 = CSRF. 두 공격의 차이를 명확히 구분하는 문제.
📄 2020 국가직 정보시스템보안 · CSRF 오답 (세션 하이재킹)★★★
CSRF(Cross Site Request Forgery)의 특징에 대한 설명으로 옳지 않은 것은?
① 사용자가 자신의 의지와는 무관하게 공격자가 의도한 행위를 특정 웹사이트에 요청하게 하는 방법이다.
② 사이트에 방문하는 사용자가 정상적인 요청이 아닌 임의의 요청을 하도록 위조하는 방법이다.
③ 서버와 클라이언트에 각각 잘못된 시퀀스 넘버를 사용하여 연결된 세션에 혼란을 줌으로써 공격자가 서버와의 연결을 획득하는 방법이다.
④ 특정 웹사이트가 사용자의 웹브라우저를 신용하는 상태를 악용하는 방법이다.
정답 ③ — "시퀀스 넘버, 세션 혼란, 서버 연결 획득" = 세션 하이재킹(Session Hijacking) 설명. CSRF는 시퀀스 번호와 무관. CSRF는 이미 맺어진 정상 세션을 '이용'하는 것이지, 세션 자체를 '탈취'하는 게 아님.
📄 2020 국가직 정보시스템보안 · XSS 공격 순서★★★
일반적인 XSS 공격 수행 과정을 순서대로 바르게 나열한 것은?
가. 사용자가 공격자의 XSS 코드에 접근한다.
나. 사용자의 시스템에서 XSS 코드가 실행된다.
다. 웹 서버가 XSS 코드가 포함된 게시판 글을 사용자에게 전달한다.
라. 공격자가 XSS 취약점이 존재하는 서버에 XSS 코드를 작성하여 저장한다.
마. XSS 코드가 실행된 결과가 공격자에게 전달되고 공격을 종료한다.
① 라→가→나→다→마
② 라→가→다→나→마
③ 라→다→가→나→마
④ 라→다→나→가→마
정답 ③ 라→다→가→나→마 — ①라: 공격자가 서버에 XSS 코드 저장. ②다: 사용자가 페이지 요청 시 서버가 코드 포함 응답 전달. ③가: 사용자가 XSS 코드에 접근(인지 못함). ④나: 사용자 브라우저에서 실행. ⑤마: 결과가 공격자에게 전달. 핵심: "서버가 전달(다)" 후 "사용자 접근(가)" 순서.
📄 2020 국가직 정보시스템보안 · SQL Injection 오답★★★
SQL 인젝션에 대한 설명으로 옳지 않은 것은?
① 공격자가 입력값을 조작하여 원하는 SQL 구문을 실행한다.
② 전송되는 패킷을 가로채어 자신이 송신자인 것처럼 패킷을 변경하여 다시 보내는 기법이다.
③ 대응 방법 중에는 사용자의 입력에 특수문자가 포함되어 있는지 검증하는 방법이 있다.
④ OWASP에서 선정한 10대 웹 어플리케이션 보안 위험으로 SQL 인젝션의 취약점이 2004·2007·2010·2013·2017년에 포함되어 있다.
정답 ② — "패킷 가로채기 + 송신자 위장 + 패킷 변경 + 재전송" = 패킷 스푸핑 또는 세션 하이재킹 설명. SQL Injection은 입력값 조작으로 SQL 쿼리를 변형하는 공격이며 패킷 가로채기와 무관.
📄 2022 국가직 정보시스템보안 · 리버스 텔넷 정의★★★

① 리버스 텔넷
② 쿠키 변조
③ 명령 삽입
④ 파일 업로드
정답 ① — "서버에서 먼저 공격자 PC로 연결" = 리버스 텔넷. 방화벽이 인바운드는 차단하지만 아웃바운드(서버→외부)는 허용하는 정책을 역이용. 주로 웹쉘 설치 후 실행됨.

(가) "../"를 이용해서 임의의 경로가 포함된 값으로 웹페이지 파라미터를 변조한 후 해당 경로의 파일 내용이 표시되는지 확인
(나) 사용자 입력값을 전달받는 게시판에 <script>alert()</script>와 같은 스크립트를 입력한 후 실행 여부 확인
(다) 인증 후 정상적으로 세션이 발행된 페이지의 정보를 취득하고 일정 시간 후에 재전송했을 때 정상 처리가 되는지 확인
① (가)경로추적 (나)CSRF (다)불충분한 인증 및 인가
② (가)경로추적 (나)XSS (다)불충분한 세션 관리
③ (가)디렉터리 인덱싱 (나)XSS (다)불충분한 인증 및 인가

정답 ② — (가): ../ 파라미터 변조 → 경로 추적(Path Traversal). (나): 스크립트 입력 후 실행 확인 → XSS. (다): 세션 정보 재전송 시 정상 처리 → 불충분한 세션 관리. 디렉터리 인덱싱은 URL 경로로 직접 접근 시 파일 목록 노출 확인.
📄 2023 국가직 정보시스템보안 · CSRF 설명★★★
CSRF에 대한 설명으로 옳은 것은?
① 취약한 컴포넌트를 악용하여 공격하면 데이터 손실이나 서버 장악 문제가 발생할 수 있는 취약점이다.
② 조작된 XPath 쿼리를 보냄으로써 비정상적인 데이터를 쿼리를 이용해 가져올 수 있는 취약점이다.
③ 적절히 보호되지 않은 쿠키를 사용하여, 쿠키 값 변조를 통한 다른 사용자로의 위장 및 권한 상승 등이 가능한 취약점이다.
④ 로그인한 사용자 브라우저로 하여금 사용자의 세션 쿠키와 기타 인증 정보를 포함하는 위조된 HTTP 요청을 전송할 수 있는 취약점이다.
정답 ④ — "로그인한 사용자 브라우저 + 세션 쿠키 포함 + 위조된 HTTP 요청 전송" = CSRF 핵심 정의. ①=취약한 컴포넌트 사용, ②=XPath Injection, ③=쿠키 변조(권한 상승).
💡 웹 취약점 공격 핵심 암기
XSS 3종Stored(저장형)·Reflected(반사형)·DOM기반. "Reflash"는 없는 유형.
XSS 공격 순서라(코드저장) → 다(서버전달) → 가(사용자접근) → 나(브라우저실행) → 마(공격자전달)
XSS vs CSRFXSS = 스크립트가 클라이언트에서 실행 / CSRF = 위조 요청이 서버로 전송
CSRF 키워드불특정 다수 / 로그인된 세션 / 사용자 의지와 무관 / 수정·삭제·송금
SQL Injection입력값 → SQL 쿼리 변형. 대응: Prepared Statement + 특수문자 필터링
SQL 대응 오답"클라이언트 측만 검증" → ✗. 반드시 서버 측 검증 필요.
경로 조작../로 파라미터 변조 → 허용 디렉터리 밖 파일 접근. 디렉터리 인덱싱(파일목록 노출)과 다름.
파일업로드→웹쉘확장자 미검증 → PHP/ASP/JSP 업로드 → 웹쉘 실행 → OS 명령어 수행
리버스 텔넷서버→공격자 방향. 아웃바운드 정책 우회. 80번 포트 이용 시 탐지 어려움.
웹쉘 대응실행금지 + 위치격리 + 형식제한 + 파일명 필터링. 4가지 모두 유효한 대응.

  • "SQL Injection: 클라이언트 측 검증만으로 해결 가능" → ✗ (서버 측 검증 필수)

  • "CSRF: 시퀀스 넘버로 세션 혼란 → 연결 획득" → ✗ (세션 하이재킹 설명)
  • "리버스 텔넷: 공격자가 서버에 접속하는 방식" → ✗ (서버→공격자 방향)
