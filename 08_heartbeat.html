Heartbeat Protocol이란?
★★★ 2025 국가직 출제
Heartbeat Protocol 정의

Heartbeat(하트비트) 프로토콜은 TLS/DTLS 연결의 생존 여부를 확인하기 위한 확장 기능. RFC 6520에 정의된 TLS/DTLS 확장 규격. 클라이언트와 서버가 연결을 유지하고 있는지 확인하기 위해 Heartbeat 요청 메시지를 주고받는 방식. 클라이언트가 임의의 데이터(페이로드)를 포함한 메시지를 서버에 보내고, 서버는 그 데이터를 그대로 되돌려주는(에코) 방식으로 동작.

Heartbeat는 이름 그대로 "심장 박동"이에요. 살아있는 사람의 심장이 계속 박동하듯, TLS 연결이 살아있는지 확인하는 신호예요. TCP 연결은 데이터를 주고받지 않아도 일정 시간 동안 유지될 수 있는데, 정말 연결이 살아있는지 아닌지 확인하는 방법이 필요해요.

Heartbeat는 SSL/TLS의 4가지 상위 프로토콜 중 하나예요. 기존 SSL의 3개(Handshake·Change Cipher Spec·Alert)에 추가된 확장 기능이에요. TLS 1.3에서도 Heartbeat 확장은 별도로 협상해서 사용할 수 있어요.

동작 방식: 클라이언트가 "나 살아있어? [페이로드 N바이트]"라는 Heartbeat Request를 보내면, 서버는 "응, 살아있어. [페이로드 그대로]"라는 Heartbeat Response를 돌려줘요. 매우 단순한 에코(Echo) 방식이에요.

TLS 프로토콜 계층 구조 — Heartbeat 위치
2025 국가직 기출 키포인트: "TLS에 속한 프로토콜 중 세션을 활성 상태로 유지하는 목적으로 사용하는 것은?" → Heartbeat Protocol. Alert=경고 알림. Application Data=데이터 전송. Connection=연결 관리(SSH 개념). 세션 유지·생존 확인 = Heartbeat.

Heartbeat 동작 흐름
원리 이해

Heartbeat Request 메시지 구조는 세 부분으로 구성돼요. 첫째 타입(Type): 요청(1) 또는 응답(2). 둘째 페이로드 길이(Payload Length): 클라이언트가 "내가 N바이트 보낼게"라고 선언하는 값. 셋째 페이로드(Payload): 실제 데이터. 서버는 페이로드 길이 필드에 적힌 만큼 메모리에서 데이터를 읽어 돌려줘야 해요.

여기서 핵심 규칙이 있어요. 서버는 페이로드 길이 필드의 값이 실제 페이로드의 크기와 일치하는지 검증해야 해요. 이 검증을 안 하면? → Heartbleed 취약점이 생겨요.

Heartbleed 취약점 완전 정리
★★★ 2024 지방직 출제
Heartbleed Vulnerability
CVE-2014-0160

Heartbleed는 TLS(Transport Layer Security)를 구현한 공개 소프트웨어인 OpenSSL 1.0.1~1.0.1f 버전에서 발견된 심각한 보안 취약점. 클라이언트가 보낸 Heartbeat 요청 메시지의 실제 페이로드 길이(Length)를 제대로 검증하지 않아 발생. 이로 인해 서버 메모리에 저장된 비밀번호, 세션 쿠키, 개인키 등 민감한 정보가 유출될 수 있음. 최대 64KB씩 반복 호출 가능.

Heartbleed의 핵심은 "길이 검증 누락"이에요. 클라이언트가 Heartbeat Request에 페이로드 길이를 65,535라고 써서 보내도, 실제 페이로드는 1바이트만 보낼 수 있어요. 취약한 서버는 이 차이를 확인하지 않고, 길이 필드에 적힌 65,535바이트만큼 메모리에서 읽어 응답해요. 1바이트 뒤에 있는 64KB는 서버 메모리에 저장된 다른 데이터예요.

왜 이름이 "Heartbleed(심장 출혈)"일까요? Heartbeat 프로토콜을 통해 서버 메모리의 내용이 피가 흘러나오듯 새어나온다는 의미에서 붙은 이름이에요. 2014년 4월에 공개됐고, 당시 전 세계 수많은 HTTPS 웹사이트가 영향을 받았어요.

2024 지방직 기출 포인트: "(가)에는 프로토콜 이름, (나)에는 검증하지 않은 것"이 들어가요. (가)=TLS(SSH가 아님!), (나)=길이(Length)(유형이 아님!)가 핵심이에요.

TLS 데이터 확장 함수 (Data-Expansion Function)
★★★ 2022 국가직 출제
데이터 확장 함수 정의

데이터 확장 함수(data-expansion function)는 비밀(secret)을 보다 길게 확장하기 위해 미리 정의된 HMAC(MD5 또는 SHA-1)을 사용하는 함수. 각 섹션은 비밀(secret)과 종자(seed)인 두 개의 HMAC을 이용. 입력: secret + seed.

데이터 확장 함수의 구조를 이해하는 게 핵심이에요. HMAC을 반복 적용해 원하는 길이의 출력을 만들어요. 입력 비밀(secret)을 HMAC으로 계속 체이닝(Chaining)해서 필요한 만큼 늘려요. 이 과정이 P_hash(secret, seed)예요.

중요한 구분: 데이터 확장 함수의 입력은 secret + seed 두 가지예요. 레이블(label)은 데이터 확장 함수에 직접 들어가는 게 아니라, label과 seed를 합쳐서(concatenation) seed로 넘겨요.

P_hash(secret, seed) — 데이터 확장 함수 구조

TLS 의사난수 함수 (Pseudorandom Function, PRF)
★★★
PRF 정의

TLS 의사난수 함수(PRF)는 MD5 또는 SHA-1을 사용하여 두 데이터 확장 함수의 결합이 되도록 정의된 함수. 입력: 비밀(secret) · 식별용 레이블(label) · 종자(seed) 3가지. 비밀은 두 개의 절반(half-secret)으로 나뉘어 각각의 데이터 확장 함수에 사용됨. 두 확장 함수의 출력을 XOR하여 최종 출력 생성.

PRF가 데이터 확장 함수와 다른 점: PRF는 입력이 3가지(secret, label, seed)예요. 데이터 확장 함수는 2가지(secret, seed). PRF는 내부적으로 데이터 확장 함수를 활용하는 상위 구조예요.

PRF 동작 구조: ① 비밀(secret)을 절반(half-secret)으로 나눔 → ② 한쪽은 MD5 기반 P_MD5, 다른 쪽은 SHA-1 기반 P_SHA-1로 확장 → ③ 두 출력을 XOR → 최종 PRF 출력. seed 자리에는 label과 seed를 연결(label || seed)해서 넣어요.

TLS PRF 구조 — 비밀을 절반으로 나눠 MD5·SHA-1 이중 처리 후 XOR

왜 틀렸냐면, 사전 마스터 비밀·'master secret'·2개 난수를 입력받는 건 PRF(의사난수 함수)예요. 데이터 확장 함수(P_hash)의 입력은 secret과 seed 두 가지뿐이에요. PRF가 내부적으로 데이터 확장 함수를 호출하는 것이지, 데이터 확장 함수가 직접 label을 받는 게 아니에요.

TLS에 대한 설명으로 옳지 않은 것은?
① TLS는 SSL을 기반으로 만들어진 인터넷 표준이다.
② TLS의 의사난수 함수(pseudorandom function)는 데이터 확장 함수(data-expansion function)를 활용한다.
③ TLS의 의사난수 함수는 비밀(secret), 식별용 라벨(label), 종자(seed)를 입력받는다.

정답 ④

④에서 "데이터 확장 함수"가 잘못됐어요. 사전 마스터 비밀·'master secret' 레이블·2개의 난수(seed)를 입력받는 것은 PRF(의사난수 함수)예요.

· 데이터 확장 함수(P_hash) 입력: secret + seed (2가지)
· PRF(의사난수 함수) 입력: secret + label + seed (3가지)

PRF가 내부적으로 데이터 확장 함수를 호출하는 구조. 마스터 비밀 생성 시 PRF(pre-master secret, "master secret", CR‖SR)를 호출하고, PRF는 이를 데이터 확장 함수에 넘겨 확장함.

①②③은 모두 옳은 설명이에요.

(가) TLS: Heartbeat Protocol은 TLS/DTLS 확장 기능(RFC 6520). SSH와 전혀 무관. OpenSSL은 TLS를 구현한 오픈소스 라이브러리.

· Alert Protocol = 오류·경고 알림 전달 (예: 인증 실패, 버전 불일치)
· Connection Protocol = SSH 개념. TLS에는 없는 구분.
· Heartbeat Protocol = TLS/DTLS 연결(세션) 생존 여부 확인 → 세션 활성 상태 유지. RFC 6520.
· Application Data Protocol = 실제 애플리케이션 데이터 전송

"세션을 활성 상태로 유지"는 Heartbeat의 핵심 목적. 심장 박동처럼 주기적으로 확인 신호를 주고받아 연결이 끊기지 않도록 함.

삽입된 악성 스크립트를 통해 사용자가 인지하지 못하는 상태에서 사용자의 권한으로 서버에 공격자가 의도한 요청을 하는 공격은?
① Slowloris
② CSRF
③ Heartbleed
④ DNS Cache Poisoning
정답 ② CSRF

이 문제는 Heartbleed를 오답으로 활용했어요. Heartbleed를 선택하지 않으려면 정확한 개념 구분이 필요해요.

· CSRF = 로그인된 사용자를 이용해 공격자가 의도한 요청을 서버에 전송. 사용자 권한 악용.
· Heartbleed = 서버 메모리 읽기 공격. 악성 스크립트 삽입과 무관. TLS 취약점.
· Slowloris = HTTP 연결 점유형 DoS 공격. DNS Cache Poisoning = DNS 캐시 변조.
