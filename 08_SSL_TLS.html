1
1994년 넷스케이프의 선택
넷스케이프

인터넷 쇼핑이 막 시작되던 1994년, 문제가 명확했어요. 신용카드 번호를 웹에서 입력하면, 그 번호가 평문으로 인터넷을 떠돌았어요. 중간에서 패킷을 가로채면 그냥 보였죠.

당시 넷스케이프(Netscape)는 웹 브라우저 시장을 사실상 독점하고 있었어요. 그들은 인터넷 상거래를 현실로 만들기 위해 반드시 이 문제를 해결해야 했어요.

"HTTP 자체를 건드리면 기존 웹 전체를 바꿔야 해요.
그냥 HTTP와 TCP 사이에 보안 유리창을 하나 끼우면 어떨까요?
HTTP는 아무것도 모르게, 우리가 중간에서 몰래 암호화·복호화하는 거예요."

이 아이디어가 SSL(Secure Sockets Layer)입니다. 1994년 넷스케이프가 설계하고 개발했어요. SSL은 TCP/IP 계층과 응용 계층 사이에 '투명한 보안 계층'을 만들어서, HTTP를 포함한 어떤 응용 프로토콜이든 SSL 위에서 돌릴 수 있게 했어요.

이것의 진짜 위대함은 범용성에 있어요. HTTP만이 아니라 FTP·SMTP·IMAP 어디든 SSL을 끼우면 바로 암호화 통신이 됩니다. 그래서 HTTPS·FTPS·SMTPS·IMAPS가 모두 같은 원리로 작동해요.

2
SSL이 해결해야 했던 3가지 문제
핵심 설계 목표

SSL이 설계 당시 해결하려 했던 세 가지 핵심 문제를 알면 SSL 전체 구조가 이해됩니다.

🎯 문제 1 — "내가 진짜 네이버에 접속한 건지 어떻게 알아?"

해결: 서버 인증(Server Authentication) — 핸드셰이크 단계에서 서버가 CA가 서명한 X.509 인증서를 보내요. 클라이언트는 이걸 검증해서 진짜 서버인지 확인해요. 중간자 공격(MITM)을 차단하는 핵심 메커니즘.

🎯 문제 2 — "전송 중에 내용이 바뀌면 어떻게 알아?"

해결: 메시지 무결성(Message Integrity) — 매 레코드마다 MAC(Message Authentication Code)을 붙여서 데이터가 중간에 변조됐는지 확인해요. 핸드셰이크에서 협의한 MAC 알고리즘으로 생성.

🎯 문제 3 — "누군가 도청해도 내용을 모르게 하려면?"

해결: 기밀성(Confidentiality) — 핸드셰이크에서 협의한 세션키(대칭키)로 모든 데이터를 암호화해요. 공개키 암호화로 세션키를 안전하게 교환 → 이후엔 빠른 대칭키 암호화 사용 (= 하이브리드 암호화 구조).

3
SSL → TLS 진화 — 인터넷 표준이 되다
역사

SSL은 넷스케이프가 만든 독점 기술이었어요. 웹이 폭발적으로 성장하면서 인터넷 표준화 기구 IETF가 이걸 표준화하기로 했어요. 그 결과물이 TLS(Transport Layer Security)입니다.

1
SSL/TLS 프로토콜 스택 구조
★★★
SSL/TLS 정의

SSL/TLS는 TCP/IP 전송 계층과 응용 계층 사이에 위치하여 종단 간 보안 서비스(기밀성·무결성·서버인증)를 제공하는 프로토콜. 넷스케이프가 1994년 최초 제안. 단 하나의 프로토콜이 아닌 두 계층의 프로토콜로 구성됨.

2
4가지 프로토콜 역할 상세
★★★
프로토콜역할핵심 동작시험 포인트
① Handshake 인증·협상·키 교환 서버 인증서 검증, 암호 알고리즘 협상, 세션키(pre-master secret) 교환, 세션 수립 모든 응용 데이터 전송 이전에 실행. 4단계 구성.
② Change Cipher Spec 암호 전환 신호 핸드셰이크로 협의한 암호 알고리즘·MAC·압축 설정을 지금부터 적용함을 상대에게 알림 1바이트로 구성된 가장 단순한 SSL 프로토콜
③ Alert 오류·경고 알림 SSL 관련 오류를 상대방에게 전달. warning(경고)과 fatal(치명적) 두 수준으로 구분 fatal → SSL 즉시 종료. 동일 세션 재사용 불가.
④ Record 기밀성·무결성 제공 단편화 → 압축 → MAC 추가 → 암호화 → 헤더 추가. 상위 3개 포함 모든 데이터 처리 기밀성 = 대칭키 암호 (MAC ≠ 기밀성)
⚠함정 — "user authentication 프로토콜이 SSL 구성 요소" (2016 정보시스템보안)
④ "user authentication 프로토콜" → ✗ 오답 (존재하지 않는 프로토콜)

SSL 4가지: Handshake / Change Cipher Spec / Alert / Record. "user authentication"은 SSH의 구성 요소. SSL에는 없는 프로토콜이에요.

2
각 단계 상세 설명
★★★
1
단계 1 — 보안 설정 (Security Capability Setup)
논리적 연결을 시작하고 보안 기능을 설정하는 단계. client_hello: 클라이언트가 지원하는 버전·랜덤값·세션ID·암호 조합·압축 방법을 서버에게 전송. server_hello: 서버가 클라이언트 제안 중 하나를 선택하여 응답. 클라이언트 랜덤 + 서버 랜덤 = 나중에 세션키 유도에 사용.
client_hello → server_hello
2
단계 2 — 서버 인증 + 키 교환
certificate: 서버가 X.509 인증서를 전송 (서버 공개키 포함). 클라이언트가 CA 서명 검증. server_key_exchange: DH 등 키 교환에 필요한 추가 파라미터 (필요 시). certificate_request: 클라이언트 인증서 요청 (선택). server_hello_done: 서버 메시지 끝, 클라이언트 응답 대기.
certificate · server_key_exchange · server_hello_done
3
단계 3 — 클라이언트 키 교환
client_key_exchange: 핵심 단계. 클라이언트가 pre-master secret(사전 마스터 비밀)을 생성하여 서버 공개키(RSA)로 암호화하거나, DH 파라미터를 전송. 이 값으로 양측이 동일한 master secret을 계산. certificate_verify: 클라이언트 인증서가 있을 경우 서명으로 소유 증명.
client_key_exchange → master secret 생성
4
단계 4 — 종료 (핸드셰이크 완료)
change_cipher_spec: "지금부터 협의된 암호 설정 적용"을 알림. finished: 지금까지 교환한 핸드셰이크 메시지 전체를 해시하여 무결성 검증. 양측이 모두 전송 → 핸드셰이크 완료. 이후 모든 통신은 세션키로 암호화.
change_cipher_spec + finished

Anonymous DH는 인증서(certificate) 없이 DH 파라미터만으로 키 교환. "인증서 메시지가 사용되지 않는 기법 = Anonymous DH" → 시험 정답. 나머지 RSA·Ephemeral DH·Fixed DH는 모두 인증서 사용.

TLS 1.2: 현재 대부분의 서비스에서 사용. SHA-256, AEAD(GCM 등) 지원 추가. ARIA(한국 표준 알고리즘)도 TLS에서 사용 가능.

3
Heartbleed 취약점 — CVE-2014-0160
★★
💀 Heartbleed (CVE-2014-0160) — 2014년 인터넷을 뒤흔든 취약점

원리: TLS의 Heartbeat 확장 기능(RFC 6520)에 있는 버그. 클라이언트가 "나 살아있어요!" heartbeat 요청을 보낼 때 페이로드 길이를 거짓으로 크게 보고하면, 서버가 실제 데이터보다 최대 64KB의 메모리를 그대로 반환해버리는 취약점.

영향: 반환된 메모리에 비밀번호, 세션 쿠키, X.509 개인키 등 민감 정보가 포함될 수 있음. OpenSSL 1.0.1 ~ 1.0.1f 버전에 존재.

교훈: SSL/TLS 자체 설계 결함이 아닌 구현(OpenSSL 라이브러리) 버그. 패치 후 키 교체·세션 무효화 필요.
