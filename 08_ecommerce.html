1
전자상거래(Electronic Commerce) 정의
★★
전자상거래 정의

전자상거래(Electronic Commerce)란 개인, 기업, 정부 간에 인터넷을 통하여 이루어지는 상거래를 의미한다. 광의로는 인터넷이나 네트워크를 이용한 관련 정보 제공, 협상, 주문, 납품, 대금 지불, 이체 등을 통해 마케팅, 판매하는 것을 말한다.

전자상거래는 단순히 인터넷으로 물건을 사고파는 것을 넘어서요. 정보 제공 → 협상 → 주문 → 납품 → 결제 → 이체까지 거래의 전 과정이 전자적으로 이루어지는 것을 포괄해요. 인터넷 쇼핑, 인터넷 뱅킹, 전자결제, 전자세금계산서, B2B 거래 모두 포함돼요.

전자상거래 환경에서는 법적·정책적·프라이버시 문제에 대한 공통된 규칙이 있어야 하고, 기술 표준과 멀티미디어 콘텐츠, 네트워크 인프라가 유지되어야 해요. 이 환경이 갖춰지지 않으면 신뢰할 수 있는 전자상거래가 이루어지기 어렵습니다.

2
전자상거래 구성요소 (환경 구조)
★★

전자상거래 환경은 크게 참여 주체, 프로세스 지원 시스템, 네트워크 인프라 세 축으로 구성돼요.

전자상거래 환경 구성도
핵심 구성요소 정리: 전자상거래에는 ① 구매자(개인·기업·정부) ② 판매자(기업·개인) ③ 프로세스 지원(결제·주문·배송·금융기관) ④ 네트워크 인프라(인터넷·상용네트워크·전용네트워크)가 반드시 필요해요.
1
전자상거래 3대 위협요소
★★★

전자상거래에 대한 위협은 크게 3가지로 분류돼요. 시스템 공격·데이터 공격·비즈니스 공격이에요. 각 유형의 정의와 특성을 정확히 구분해야 해요.

⚔ ① 시스템 공격
인터넷에 연결된 서버를 불법으로 침입하여 컴퓨터를 오작동시키거나, 데이터 유출·삭제·변경하는 공격.
이런 공격을 막기 위해 방화벽 같은 보안 도구를 사용한다.
외부 공격으로 인한 손실은 내부 직원에 의한 손실보다 불법적인 경우가 더 많다.
💾 ② 데이터 공격
전자상거래에서 데이터 공격은 저장된 데이터와 전송 중인 데이터로 구분할 수 있음.
저장 시에도 암호화를 통한 보호가 필요하다.
전송 중인 데이터에 대한 공격을 막기 위해 기밀성·무결성 등에 대한 보호가 요구된다.
🏢 ③ 비즈니스 공격
전자상거래에서는 익명성이라는 특징 때문에 거래하는 제3자가 누구인지 알 수 없을 수 있음.
오프라인에서만 일어날 수 있는 공격이 전자적 환경에서도 일어날 수 있음.
법적인 장치, 전자적 시스템 외적인 합의가 이루어져야 한다.
2
인터넷 뱅킹 시스템 위험요소 5종
★★★ 2022 경찰간부 출제

전자상거래·인터넷 뱅킹 시스템의 위험요소는 5가지 구성요소별로 분류돼요. 각 요소에 해당하는 구체적인 위협 이름까지 매핑해서 외워야 해요. 2022 경찰간부 기출에서 요소별 위협 이름 매핑이 출제됐어요.

구성요소위협 요소설명
① 클라이언트 보안 Virus, Trojan Horse 사용자 단말기(PC·스마트폰)에 악성코드 감염. 키로거, 화면 캡처 등으로 인증정보 탈취.
② 처리요소 보안
(거래처리)		 Weak Encryption, Server Spoof 거래 전송 구간의 암호화 취약 또는 서버 위장(Spoofing). 전송 중 데이터 도청·위변조.
③ 서버 보안 Unix/NT Security Holes, Improper Administration 서버 OS 취약점 악용, 관리 소홀(패치 미적용, 불필요 서비스 운영). 서버 직접 침입.
④ 애플리케이션 보안 Poor Programming, Weak Authentication 불안전한 코드(SQL Injection·BOF 등), 취약한 인증 로직. 앱 자체 취약점.
⑤ 내부적 보안 Virus & Trojan Horse,
Unix/NT Security Holes,
Improper Administration		 내부자에 의한 위협. 클라이언트+서버 위협이 내부에서 복합적으로 발생할 수 있음.
⚠함정 ★★★ — 거래처리 vs 애플리케이션 위협 혼동 (2022 경찰간부)
③ 거래처리 보안에서의 위협요소 — Poor Programming, Weak Authentication → ✗ 오답

Poor Programming, Weak Authentication은 애플리케이션 보안(④)의 위협요소예요. 거래처리(②) 보안 위협은 Weak Encryption, Server Spoof이에요. 처리 중 전송 구간의 암호화 문제와 서버 위장이 핵심.

내부적 보안(⑤) = Virus & Trojan Horse + Unix/NT Security Holes + Improper Administration → 클라이언트(①)와 서버(③) 위협이 모두 포함되는 가장 넓은 범위.

1
전자상거래 보안 요구사항 7가지
★★★

전자상거래에서는 일반 정보보호의 CIA(기밀성·무결성·가용성)를 넘어 거래의 특성에 맞는 7가지 보안 요구사항이 필요해요. 특히 신분 확인 및 인증이 모든 보안 기능의 전제조건이에요.

1
신분 확인 및 인증 (Identification & Authentication)
거래하는 당사자를 서로 확인하고 있어야 한다. 인증 수단으로는 ID/패스워드, 암호화 기반 인증서(PKI) 등이 포함된다.
모든 보안 기능(접근통제·기밀성·무결성·부인방지·프라이버시 관리)의 전제조건이 되는 가장 기본적이고 중요한 보안 요건이다.
가장 기본적이고 중요한 전제조건 ★
2
기밀성 (Confidentiality)
저장되어 있거나 전송 중인 정보를 인가되지 않은 자가 접근할 수 없도록 해야 한다.
특히 결제 정보, 주문 정보는 더욱 중요하게 보호되어야 한다. 암호화를 통해 제3자가 접근할 수 없도록 보호한다.
3
무결성 (Integrity)
저장되어 있거나 전송 중인 정보가 인가하지 않은 자에 의해 변조되거나 삭제되지 않도록 해야 한다.
무결성 보장을 위해 전자서명·해시 함수를 이용한다.
4
가용성 (Availability)
정당한 사용자에게 서비스가 중단되지 않고 원활하게 제공될 수 있도록 해야 한다.
DoS·DDoS 공격 같은 서비스 거부 공격으로부터 보호되어야 하며, 장애 발생 시 대처할 수 있는 복구 능력이 있어야 한다.
5
거래 부인 방지 (Non-repudiation)
거래 당사자 간에 정보 제공 및 수신에 대한 부인을 방지하는 것이다.
전자서명을 이용한 법률 분쟁에서 매우 중요한 역할을 하며, 전자서명에 의해 해결할 수 있다.
6
접근 제어 (Access Control)
정보에 대한 변경 및 모든 접근에 대해 명백히 제어함으로써 인가되지 않은 접근을 차단할 수 있도록 하는 것이 요구된다.
서버에서는 접근 통제 시스템을 운영하며, 클라이언트에서는 암호화를 강화하여 통제 수준을 높일 수 있다.
7
보안 관리 (Security Management)
모든 보안 기능에 대한 체계적인 관리가 요구되며, 이를 위해 보안 정책·기준·절차의 제정, 보안 기술에 대한 교육이 이루어져야 한다.
또한 보안 사고에 대한 처리 절차와 보안 감사가 이루어져야 한다.
요구사항핵심 키워드주요 보안 기술
신분 확인 및 인증거래 당사자 확인 · 모든 보안의 전제ID/PW, PKI 인증서
기밀성인가되지 않은 접근 차단 · 결제·주문 보호암호화
무결성변조·삭제 방지전자서명, 해시 함수
가용성서비스 중단 없이 원활 제공DDoS 대응, 이중화
부인 방지거래 사실 부인 방지 · 법률 분쟁전자서명
접근 제어인가되지 않은 접근 차단방화벽, 접근통제 목록
보안 관리정책·기준·절차 제정 · 보안 감사ISMS, 보안 감사
1
SET (Secure Electronic Transaction) 개요
★★★
SET 정의

SET(Secure Electronic Transaction)는 비자(VISA)와 마스터카드(MasterCard)가 합동으로 개발한 인터넷상에서 신용카드를 이용한 상품 구매 시 안전한 대금결제 처리를 위한 프로토콜. RSA 암호화와 인증 기술을 이용하여 네트워크상에서 민감한 개인금융 정보를 안전하게 전송하기 위해 개발됨. 전자봉투(Digital Envelope)와 이중서명(Dual Signature)을 핵심 기법으로 사용.

SET가 일반 SSL/HTTPS 결제와 다른 결정적인 차이는 이중서명(Dual Signature)이에요. 카드소지자의 주문 정보는 상점에만, 지불 정보(카드번호 등)는 은행에만 보여요. 상점이 카드번호를 알 수 없고, 은행이 주문 내용을 알 수 없어요. 이중 서명 기법으로 프라이버시를 보호하면서도 거래의 정당성을 증명해요.

SET의 또 다른 특징은 PKI(공개키 기반구조) 전면 도입이에요. 모든 참여자(카드소지자·상점·은행·게이트웨이)가 각자의 인증서를 보유하고, CA(인증기관)가 신원을 보증해요. 신용카드 소지자에게는 전자지갑(Wallet) 소프트웨어가 필요하고, 상점에도 SET 전용 소프트웨어가 필요해요.

2
SET 6대 참여 주체
★★★
① 카드소지자 (Cardholder)
카드를 발급받아 인터넷에서 상품·서비스를 구매하는 사람. 전자지갑(Wallet) 소프트웨어 필요. 인증서 보유.
② 가상 상점 (Merchant)
인터넷 상에서 카드소지자에게 상품·서비스를 판매하는 주체. SET 전용 소프트웨어 필요. 결제 게이트웨이와 연결.
③ 카드 발급사 (Issuer)
카드소지자에게 카드를 발급하는 금융기관. 계좌 개설·관리 담당.
④ 지불 처리 은행 (Acquirer)
상점(가맹점)과 계약을 맺고 거래를 처리하는 금융기관. 상점의 은행.
⑤ 지불 게이트웨이 (Payment Gateway)
처리 은행 또는 제3자에 의해 운영. 요청한 지급 승인 및 처리를 하는 카드 지불 네트워크로의 통로.
⑥ 인증 기관 (CA)
SET 참여자에게 인증서를 발행하는 기관. 각 주체의 신원 보증.

"SET는 오프라인상에서 금융거래 안전성을 보장하기 위한 시스템이다" → ✗ 오답

SET는 온라인(인터넷) 신용카드 거래 보안 프로토콜. 오프라인이 아님.

"신용카드 사용을 위해 상점에서 소프트웨어를 요구하지 않는다" → ✗ 오답

SET는 카드소지자(전자지갑)와 상점 모두에게 전용 소프트웨어 필요. 상점도 SET 소프트웨어 설치 필요.

"SET 참여자들이 신원을 확인하지 않고 인증서를 발급한다" → ✗ 오답

CA(인증기관)가 SET 참여자들의 신원을 확인한 후 인증서 발급. 신원 확인이 전제됨.

"RSA 암호화를 사용하여 속도가 빠르다" → ✗ 오답

RSA는 공개키 암호화로 속도가 느림. SET는 RSA 사용으로 처리 속도가 느린 것이 단점 중 하나.

1
이중서명 (Dual Signature) 원리
★★★ 경찰간부 출제
이중서명 정의

이중서명(Dual Signature)은 SET의 핵심 보안 기법. 상점은 카드소지자의 지불 정보(카드번호 등)를 알 수 없고, 은행은 카드소지자가 구매한 물건(주문 내용)을 알 수 없지만, 두 정보가 연결되어 있음을 증명할 수 있게 하는 서명 방식. 프라이버시 보호와 거래 무결성을 동시에 달성.

이중서명의 원리를 단계적으로 살펴볼게요. 카드소지자는 두 가지 메시지를 만들어요. 주문 정보(OI, Order Information)와 지불 정보(PI, Payment Information)예요. 이 두 메시지를 직접 합치지 않고 해시로 연결해요.

구체적인 과정: ① 주문 정보의 해시(OIMD) 생성 → ② 지불 정보의 해시(PIMD) 생성 → ③ OIMD와 PIMD를 연결(||)하여 다시 해시 → ④ 이 결합 해시를 카드소지자의 개인키로 서명 = 이중서명(DS). 이 이중서명 하나로 두 메시지가 연결됨을 증명해요.

이중서명(Dual Signature) 생성 및 검증 구조

이중서명의 검증 방법: 상점은 OI를 알고 있고 PIMD(지불정보 해시)를 받아요. 상점은 Hash(OIMD‖PIMD)를 계산하고, 카드소지자의 공개키로 DS를 검증해서 일치하면 주문과 지불이 같은 거래임을 확인해요. 은행도 마찬가지로 PI를 알고 OIMD(주문정보 해시)를 받아 검증해요. 실제 내용은 전혀 모르지만 연결된 거래임을 증명하는 것이 이중서명의 핵심이에요.

SET 암호화 절차 (전자봉투 + 이중서명)
★★ 심화

SET에서 카드소지자가 상점에 구매 요청을 보낼 때의 암호화 절차를 구체적으로 살펴볼게요. 하이브리드 암호 방식(대칭키+공개키)과 이중서명이 결합돼요.

단계암호화 절차목적
①주문 정보(OI)를 대칭키(세션키)로 암호화OI 기밀성 보호
②대칭키를 상점의 RSA 공개키로 암호화 → 전자봉투세션키를 상점에 안전하게 전달
③지불 정보(PI)를 별도 대칭키로 암호화PI 기밀성 보호
④PI용 대칭키를 지불 게이트웨이의 RSA 공개키로 암호화PI는 은행(게이트웨이)만 복호화 가능
⑤Hash(OI)와 Hash(PI)를 연결 후 해시 → 카드소지자 개인키로 서명 = 이중서명(DS)두 메시지가 같은 거래임을 증명
⑥암호화된 OI + OI용 전자봉투 + 암호화된 PI + PI용 전자봉투 + DS + 카드소지자 인증서를 상점에 전송상점은 OI만 해독, PI는 해독 불가

복호화 절차: 상점은 ① 자신의 RSA 개인키로 전자봉투 복호화 → 대칭키 획득 → ② 대칭키로 OI 복호화 → ③ 카드소지자 공개키로 DS 검증. 지불 게이트웨이(은행)는 ① 자신의 RSA 개인키로 전자봉투 복호화 → PI용 대칭키 획득 → ② 대칭키로 PI 복호화 → ③ DS 검증. 두 주체 모두 서로의 정보에 접근하지 못해요.

FDS 정의

FDS(Fraud Detection System)는 전자금융거래에서 사용되는 단말기 정보, 접속 정보, 거래 내용 등을 종합적으로 분석하여 의심 거래를 탐지하고 이상금융거래를 차단하는 시스템. 보안 프로그램에서 방지하지 못하는 전자금융사기에 대한 이상 거래를 탐지하여 조치할 수 있도록 지원.

FDS는 기존 보안 프로그램(백신, 방화벽 등)이 탐지하지 못하는 정상적인 인증을 통과한 비정상 거래를 탐지하는 게 특징이에요. 예를 들어 평소에 국내에서만 사용하던 계좌에서 갑자기 해외에서 대규모 이체가 발생하면, 인증 자체는 통과했어도 FDS가 이상 거래로 탐지할 수 있어요.

FDS 분석 항목: 단말기 정보(기기 고유값·OS 버전), 접속 정보(IP 주소·접속 시간·위치), 거래 정보(금액·빈도·수신 계좌), 행동 패턴(평소 거래 패턴과의 비교) 등을 실시간으로 종합 분석해요.
