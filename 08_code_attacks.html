1
프로그램이 예상보다 큰 입력을 받으면 무슨 일이 생길까?

1988년 11월, 세계 최초의 대규모 인터넷 웜이 퍼졌어요. 모리스 웜(Morris Worm)이에요. 당시 인터넷에 연결된 컴퓨터의 약 10%가 감염됐어요. 이 웜이 이용한 핵심 기법 중 하나가 버퍼 오버플로우(Buffer Overflow)였어요. 30년이 지난 지금도 버퍼 오버플로우는 여전히 중요한 공격 기법이에요.

버퍼 오버플로우의 원리는 단순해요. 프로그램이 메모리에 데이터를 저장할 공간(버퍼)을 만들어 두는데, 그 공간보다 더 큰 데이터를 집어넣으면 어떻게 될까요? 데이터가 버퍼를 넘쳐서 인접한 메모리 영역을 덮어써요. 그 인접한 메모리에 "다음에 실행할 함수의 주소(반환 주소, RET)"가 있다면, 공격자는 그 주소를 자신이 원하는 코드의 주소로 바꿀 수 있어요. 프로그램이 함수를 끝내고 돌아갈 때 공격자의 코드가 실행돼요.

왜 이런 일이 C/C++ 프로그램에서 자주 생길까요? C언어의 strcpy(), gets(), scanf() 같은 함수들은 입력 데이터의 길이를 검사하지 않아요. 배열 경계도 확인하지 않아요. 컴파일러가 경고를 보내도 개발자들이 무시하는 경우가 많았어요. 이 "데이터 길이에 대한 불명확한 정의"가 버퍼 오버플로우의 근본 원인이에요.

포맷 스트링 공격은 버퍼 오버플로우와 형제 같은 공격이에요. C언어의 printf() 함수는 형식 지정자(%s, %n 등)를 해석하는 기능이 있는데, 사용자 입력을 형식 문자열로 그대로 사용하면 공격자가 이 기능을 악용할 수 있어요. 이건 "데이터 형태에 대한 불명확한 정의"에서 발생해요. 두 공격의 근본 원인은 비슷하지만 공격 원리는 달라요.

코드 공격의 공통 근본 원인:
버퍼 오버플로우 = 데이터 길이에 대한 불명확한 정의
포맷 스트링 = 데이터 형태에 대한 불명확한 정의
레이스 컨디션 = 프로세스 간 자원 경쟁을 이용한 공격
2
스택이 뭔지 알아야 공격을 이해할 수 있다

프로그램이 실행되면 메모리가 여러 영역으로 나뉘어요. 텍스트(코드) 영역에는 실행할 명령어가 있고, 데이터 영역에는 전역 변수가, 스택(Stack)에는 지역 변수와 함수 호출 정보가, 힙(Heap)에는 동적으로 할당된 메모리가 저장돼요.

함수가 호출될 때 스택에는 순서대로 쌓여요. 먼저 함수에 전달할 인자, 그다음 반환 주소(RET — 함수가 끝나면 어디로 돌아갈지), 그리고 이전 프레임 포인터(SFP), 마지막으로 지역 변수들이에요. 이 구조가 핵심이에요. 지역 변수(버퍼)에 오버플로우가 생기면 위로 올라가면서 SFP, 그다음 RET을 덮어써요. RET이 바뀌면 함수 종료 후 공격자가 원하는 코드로 점프해요. 이게 스택 버퍼 오버플로우의 원리예요.

힙(Heap) 오버플로우는 조금 달라요. 힙은 malloc()으로 런타임에 동적으로 할당하는 영역이에요. 프로그램이 실행될 때까지 얼마나 필요한지 알 수 없는 가변적인 양의 데이터를 저장해요. 프로그램들에 의해 할당됐다가 회수되는 작용이 반복돼요. 힙 오버플로우는 힙 청크의 메타데이터나 포인터를 덮어써서 공격해요.

1
버퍼 오버플로우 정의 및 원리
★★★ 최빈출
버퍼 오버플로우 정의

버퍼 오버플로우(Buffer Overflow)는 데이터 길이에 대한 불명확한 정의로 인해 발생하는 취약점. 프로그램 메모리 버퍼를 넘치게 해서 인접한 메모리 영역(반환 주소 등)을 덮어씀으로써 프로그램의 실행 흐름을 조작. 스택 기반 오버플로우와 힙 기반 오버플로우 2종. C언어의 경계 검사 없는 함수(gets(), strcpy(), scanf() 등)가 주요 원인.

스택 버퍼 오버플로우 동작 원리 — 정상 vs 공격 상황

스택 버퍼 오버플로우 공격 절차를 단계적으로 살펴보면, 공격자는 먼저 쉘코드(Shellcode)를 포함한 큰 데이터를 준비해요. 이 데이터를 입력하면 버퍼를 넘쳐 SFP와 RET까지 덮어써요. RET에는 쉘코드가 있는 메모리 주소(또는 NOP 썰매 앞부분)를 넣어요. 함수가 return 할 때 RET에 저장된 주소로 점프하면서 쉘코드가 실행돼요. 쉘코드가 일반 사용자 권한이면 일반 쉘이, SetUID가 설정된 root 소유 프로그램이면 root 권한의 쉘이 실행돼요.

덮어쓰는 것힙 청크 메타데이터 · 포인터
결과포인터 조작 → 임의 코드 실행

힙 영역 식별 문제 핵심: "프로그램이 실행될 때까지 알 수 없는 가변적인 양의 데이터를 저장하기 위해 프로세스가 사용할 수 있도록 예약된 메인 메모리 영역으로, 프로그램들에 의해 할당됐다가 회수되는 작용이 되풀이된다" → 힙(Heap). 스택은 LIFO 방식으로 함수 호출 시 자동 관리됨.

2020년 국회직 기출에서는 malloc()으로 힙에 버퍼를 할당하고 strcpy()로 경계 검사 없이 복사하는 코드를 보여주며 "어떤 공격에 취약한가?"를 물었어요. 키포인트는 메모리 할당 위치예요. 지역 변수(자동 할당) = 스택, malloc()(동적 할당) = 힙. 이 구분이 시험에 자주 나와요.

"gets()는 버퍼 오버플로우 공격에 취약하지 않은 함수이다" → ✗ 오답 (2016 국가직)

gets()는 버퍼 오버플로우에 가장 취약한 함수. 입력 길이를 전혀 검사하지 않음. C 표준에서도 deprecated(사용 지양). 반드시 fgets()로 대체해야 함.

"버퍼 오버플로우 방어: 문자열 조작 같은 표준 라이브러리를 안전한 것으로 교체한다 → strcat()로 교체" → ✗ 오답 (2015 지방직)

strcat()도 길이 검사 없이 동일하게 취약. '안전한 것으로 교체'의 의미는 strncat()처럼 길이 제한 버전으로 교체하는 것. 표준 라이브러리로 단순 교체가 아님.

1
셸코드 (Shellcode)
★★★
셸코드 정의

셸코드(Shellcode)는 쉘(Shell)을 실행하기 위한 스크립트로, 주입된 코드가 실행될 때 시스템 쉘을 획득하게 해주는 기계어 코드. 버퍼 오버플로우 공격에서 버퍼에 삽입되어 RET이 가리키는 코드. 쉘이 실행되면 해당 프로세스의 모든 명령을 실행할 수 있음. 일반 사용자 권한이면 일반 쉘, SetUID가 설정된 root 소유 프로그램에서 실행되면 root 권한의 쉘이 실행됨.

셸코드는 일반 코드와 다른 특성이 있어요. 첫째, NULL 바이트(\0)가 없어야 해요. strcpy()처럼 NULL을 문자열 끝으로 인식하는 함수를 통해 주입되기 때문이에요. NULL이 중간에 있으면 복사가 거기서 끊겨요. 둘째, 크기가 작아야 해요. 버퍼 안에 들어가야 하니까요. 셋째, 실행 가능한 기계어 코드여야 해요. 어셈블리어로 직접 작성하는 경우가 많아요.

실제 공격에서는 셸코드 앞에 NOP 썰매(NOP Sled)를 붙여요. NOP(No Operation, \x90)는 아무것도 하지 않고 다음 명령으로 넘어가는 명령어예요. NOP 썰매가 길면 RET가 정확히 셸코드 시작 주소를 가리키지 않아도, NOP 범위 어딘가만 가리키면 결국 셸코드까지 실행돼요. 정확한 주소를 알기 어려울 때 쓰는 기법이에요.

일반적인 환경에서 쉘을 직접 실행할 기회는 많지 않아요. 보통 사용자라면 시스템 파일 수정이나 프로그램 설치에 상당한 제한이 있어요. 하지만 관리자(root) 권한이라면 어떤 제한도 없어요. 그래서 버퍼 오버플로우를 이용한 공격의 최종 목표는 root 권한 쉘 획득이에요. 이때 SetUID가 핵심 역할을 해요.

2
SetUID — root 권한 쉘 획득의 열쇠
★★★
SetUID 정의

SetUID는 실행 파일에 설정하는 특수 권한으로, 실행하는 동안 파일 소유자의 권한으로 실행되는 메커니즘. SetUID가 설정된 파일의 실행 권한 자리에 's' 또는 'S'가 표시됨. 각각 UID=0, GID=0인 root가 소유하며 SetUID가 설정된 프로그램은 실행 시 누가 실행하든 root 권한으로 동작. 버퍼 오버플로우와 결합 시 root 쉘 획득 가능.

SetUID가 왜 필요한지 먼저 이해해야 해요. /usr/bin/passwd 명령어를 생각해보세요. 일반 사용자가 자신의 비밀번호를 바꾸려면 /etc/shadow 파일을 수정해야 해요. 그런데 /etc/shadow는 root만 쓸 수 있어요. 어떻게 해결할까요? passwd 프로그램에 SetUID를 설정해요. 일반 사용자가 실행해도 root 권한으로 동작해서 /etc/shadow를 수정할 수 있어요.

이 편리한 기능이 공격의 표적이 돼요. root 소유 + SetUID가 설정된 프로그램에서 버퍼 오버플로우가 발생하면, 공격자가 주입한 셸코드가 root 권한으로 실행돼요. 일반 사용자가 공격을 실행해도 root 쉘을 얻게 되는 거예요. 이게 레이스 컨디션 공격의 필요 조건에도 SetUID가 포함되는 이유예요.

리눅스에서 SetUID 설정 확인: ls -l 명령으로 소유자 실행 권한 자리에 s(소문자)가 있으면 실행 권한이 있고 SetUID 설정됨. S(대문자)는 실행 권한 없이 SetUID만 설정된 상태.

SetUID + 버퍼 오버플로우 → root 쉘 획득 공격 체인
1
포맷 스트링 공격 (Format String Attack)
★★★
포맷 스트링 공격 정의

포맷 스트링 공격은 데이터 형태에 대한 불명확한 정의로 발생하는 취약점. C언어의 printf() 같은 출력 함수에서 사용자 입력을 포맷 문자열로 직접 사용할 때 발생. 포맷 지정자(%x, %n 등)를 악용하여 메모리 내용을 읽거나 임의의 메모리 주소에 값을 쓸 수 있음. 버퍼 오버플로우와 같이 취약 함수 이용이라는 공통점이 있지만 공격 원리는 다름.

포맷 스트링 취약점의 핵심을 이해하려면 printf()의 동작 방식을 알아야 해요. printf("%d", num)처럼 쓰면 형식 지정자 %d와 인자 num이 대응해요. 그런데 개발자가 실수로 printf(user_input)처럼 사용자 입력을 형식 문자열 자리에 직접 넣으면 어떻게 될까요?

공격자가 %x%x%x%x를 입력하면 printf는 스택에서 값을 4개 읽어 16진수로 출력해요. 대응하는 인자가 없어도 스택을 계속 읽어요. 이를 통해 스택 메모리 내용을 덤프할 수 있어요. main() 스택 프레임의 값들이 노출되죠. 더 위험한 건 %n이에요. %n은 "지금까지 출력한 바이트 수를 해당 포인터가 가리키는 메모리에 저장"해요. 즉 메모리에 값을 쓸 수 있어요. RET 주소에 쉘코드 주소를 써넣으면 버퍼 오버플로우와 유사한 효과를 얻을 수 있어요.

포맷 스트링 공격은 런타임(실행 시점)에 발생해요. 컴파일 시점에는 탐지되지 않아요. 일부 컴파일러가 경고를 주지만 오류는 아니에요. 2025 법원직 기출에서 "컴파일 시점에 발생하며, 실행 시에는 영향을 미치지 않는다"는 선지가 오답으로 출제됐어요.

/* 취약한 코드 */
char user_input[256];
fgets(user_input, 256, stdin);
printf(user_input); /* ← 위험! 사용자 입력을 형식 문자열로 직접 사용 */

/* 공격자가 "%x%x%x%x%n" 입력 시 → 스택 메모리 읽기 + 쓰기 가능 */

/* 안전한 코드 */
printf("%s", user_input); /* ← 명시적 형식 지정자 사용 */
포맷 지정자동작공격 활용

2
레이스 컨디션 공격 (Race Condition)
★★
레이스 컨디션 정의

레이스 컨디션(Race Condition)은 프로세스 간의 자원 경쟁을 이용한 공격. 여러 프로세스가 같은 자원에 동시에 접근할 때 발생하는 경쟁 상태를 악용. SetUID가 설정된 root 소유 프로그램이 임시 파일을 생성할 때, 파일 생성 직후 ~ 사용 직전 사이의 짧은 틈을 이용해 임시 파일을 공격자가 원하는 파일의 심볼릭 링크로 교체하는 방식이 대표적.

레이스 컨디션 공격의 필요 조건 3가지를 기억하세요. ①파일의 소유자가 root여야 하고, ②SetUID 비트가 설정되어 있어야 하고, ③바로 생성되는 임시 파일의 이름을 알고 있어야 해요. 이 세 조건이 갖춰져야 공격이 가능해요.

공격 원리를 구체적으로 보면, root 소유 SetUID 프로그램이 실행 중에 임시 파일 /tmp/tempfile을 생성해요. 공격자는 그 순간을 노려요. 프로그램이 파일 이름을 결정하고 실제로 파일을 열기 직전 사이에, 공격자가 /tmp/tempfile을 /etc/passwd에 대한 심볼릭 링크로 바꿔요. 그러면 프로그램이 /tmp/tempfile에 쓴다고 생각하지만 실제로는 /etc/passwd를 수정하게 돼요. 이게 레이스 컨디션이에요.

레이스 컨디션 대응책은 임시 파일 이름을 예측 불가능하게 만들거나(mkstemp() 사용), 파일 생성 시 원자적(Atomic) 연산을 사용하거나, O_CREAT | O_EXCL 플래그로 파일이 없을 때만 생성하도록 하는 거예요.

레이스 컨디션은 프로세스 간 자원 경쟁이 핵심이에요. "버퍼 오버플로우는 프로세스 간의 자원을 경쟁하여 발생하는 공격에 이용된다"는 설명은 오답이에요(2016 국가직). 자원 경쟁 = 레이스 컨디션.

필요 조건 ①
파일 소유자가 root여야 함
필요 조건 ②
SetUID 비트가 설정되어 있어야 함
필요 조건 ③
바로 생성되는 임시 파일 이름을 알고 있어야 함
공격 핵심
파일 생성 ~ 사용 사이 틈에 심볼릭 링크로 교체
1
버퍼 오버플로우 4가지 대응책
★★★ 최빈출

버퍼 오버플로우 대응책은 시험에서 거의 매년 출제돼요. 4가지를 이름·원리·한계까지 모두 이해해야 해요. 특히 Stack Guard(스택가드)의 Canary(카나리아)와 ASLR은 단골 출제 포인트예요.

1
Non-Executable Stack (NX / DEP)
스택 영역에 실행 권한(execute)을 제거하여, 스택에 적재된 쉘코드를 실행할 수 없도록 하는 방법. CPU/OS 수준에서 지원. 스택에서 코드 실행 자체를 막음.
한계: Return-to-libc 공격(ret2libc)으로 우회 가능 — 이미 실행 권한이 있는 libc의 system() 함수를 RET에 넣는 방식.
스택 실행 권한 제거 / ret2libc 우회 가능
2
Stack Guard (스택가드) — Canary 값 이용
GCC 컴파일러 확장. 함수 진입 시 이전 프레임 포인터(SFP) 앞에 Canary 값을 삽입하고, 함수 종료 시 Canary 값이 변경됐는지 확인. 변경됐으면 버퍼 오버플로우가 발생한 것으로 판단하고 프로그램을 강제 종료.
Canary 종류: Random Canary(무작위), Null Canary(0 포함), Terminator Canary(문자열 종료 문자 포함).
★★★ SFP 앞 Canary 삽입 → 변조 감지 → 프로그램 종료
3
Stack Shield (스택실드)
함수 호출 시 반환 주소(RET)를 Global RET이라는 별도의 특수 스택에 복사해두고, 함수 종료 시 실제 RET와 Global RET의 값을 비교. 다르면 공격으로 판단하고 종료.
Stack Guard와 차이: Stack Guard는 SFP와 RET 사이 Canary 값으로 감지. Stack Shield는 RET 원본 값을 별도 저장하여 직접 비교.
RET 원본 별도 저장 → 리턴 시 비교
4
ASLR (Address Space Layout Randomization)
프로그램 실행 시마다 스택·힙·라이브러리·데이터 영역의 주소를 무작위로 배치하여 공격자가 정확한 주소를 예측할 수 없게 함. Linux 커널 2.6.12 이후 적용됨. 현재 가장 널리 사용되는 방어 기법.
효과: 쉘코드 주소나 libc 주소를 고정값으로 하드코딩한 공격이 무력화됨. 한계: 32비트 시스템에서는 주소 공간이 작아 brute-force 우회 가능.
★★★ 실행마다 주소 랜덤화 / Linux 2.6.12 이후
Stack Guard 동작 원리 — Canary 값으로 RET 변조 탐지
대응책핵심 원리한계·우회시험 키워드
NX/DEP스택 실행 권한 제거ret2libc 공격으로 우회Non-Executable Stack, execute 권한 제거
Stack GuardSFP 앞에 Canary 삽입, 종료 시 검사Canary 값을 알면 우회 가능★ 프레임포인터·카나리아·반환주소

⚠함정 ★★★ — 스택가드 Canary 위치 (2016 국가직, 2024 국가직)
스택가드에서 ㉠=프레임포인터, ㉡=카나리아, ㉢=반환주소 → ✓ 정답 ③ (2016 국가직)

스택가드는 함수 진입 시 이전 프레임 포인터(SFP) 앞에 카나리아(Canary) 값을 삽입하고 함수 종료 시 카나리아 값이 변경됐는지 확인한 후 제어를 반환 주소(RET)로 이동. ㉠=프레임포인터, ㉡=카나리아, ㉢=반환주소 순서 정확히 외울 것.

(가) 컴파일러가 함수 호출 시 ret 앞에 canary 값 주입 → 종료 시 변조 확인 = 스택 가드 (2024 국가직 정답 ④)

스택 실드(Stack Shield)와 혼동 주의: 스택 가드 = Canary 삽입. 스택 실드 = RET을 별도 공간에 복사 후 비교.

④ %n 토큰을 활용하면 공격자는 메모리의 내용을 변경할 수 있다.
정답 ① — 포맷 스트링 공격은 1999~2000년경 발견. 버퍼 오버플로우는 1988년 모리스 웜에서 이미 활용. 즉 포맷 스트링이 버퍼 오버플로우보다 훨씬 이후에 발견됨. "이전"이 아니라 "이후".
📄 2024 국가직 정보시스템보안 · 스택가드 vs 포맷 스트링★★★
(가) 컴파일러가 함수 호출 시 ret 앞에 canary 값을 주입하고 종료 시 변조 확인 →
(나) 데이터의 형태와 길이에 대한 불명확한 정의 중 데이터 형태에 대한 불명확한 정의로 발생하는 공격
① (가)스택 실드, (나)버퍼 오버플로
② (가)스택 실드, (나)포맷 스트링
③ (가)스택 가드, (나)버퍼 오버플로
④ (가)스택 가드, (나)포맷 스트링
정답 ④ — (가): "canary 값 주입 → 변조 확인" = 스택 가드(Stack Guard). 스택 실드는 RET을 별도 공간에 복사하여 비교. (나): 데이터 형태 불명확 = 포맷 스트링. 데이터 길이 불명확 = 버퍼 오버플로우.
📄 2025 법원직 9급 · 포맷 스트링 = 컴파일 시점 발생 오답★★
포맷 스트링 공격에 관한 설명 중 가장 옳지 않은 것은?
① 포맷 지정자를 악용하여 메모리 내용을 유출하거나 조작하는 공격 방법이다.
② C언어 계열에서 printf()와 같은 출력 함수에 사용자 입력을 포맷 문자열로 직접 전달할 때 발생할 수 있다.
③ 컴파일 시점에 발생하며, 실행 시에는 영향을 미치지 않는다.
④ 사용자 입력을 포맷 문자열로 받지 않고 명시적인 서식 지정자를 사용하면 방지할 수 있다.
정답 ③ — 포맷 스트링 공격은 런타임(실행 시점)에 발생. 컴파일 시점에는 탐지되지 않음(일부 컴파일러가 경고를 주지만 오류가 아님). 실행 중 사용자 입력이 포맷 문자열로 해석될 때 발생.
📄 2025 군무원 9급 · 시스템 해킹 취약점 아닌 것★★
시스템 해킹에 사용되는 취약점으로 가장 적절하지 않은 것은?
① 패스워드 터미널
② 버퍼 오버플로
③ 동적 포맷 스트링
④ 레이스 컨디션
정답 ① — "패스워드 터미널"은 시스템 해킹 취약점 유형이 아닌 존재하지 않는 용어. 시스템 해킹 3대 취약점 = 버퍼 오버플로우 / 포맷 스트링 / 레이스 컨디션.
📄 2025 국회직 9급 · 포맷 스트링 공격 그림 식별★★
다음 그림과 같이 수행하는 공격의 명칭은? (스택 메모리 덤프 + %n으로 메모리 조작 흐름도)

② 백도어 공격

④ 레이스 컨디션 공격

정답 ③ — %x로 스택 내용 출력, %n으로 메모리 값 조작하는 흐름 = 포맷 스트링 공격. 버퍼 오버플로와 다른 점: 포맷 스트링은 길이가 아닌 형식 지정자(%)를 악용.
