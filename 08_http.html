1
HTTP (HyperText Transfer Protocol) 정의
★★★
HTTP 정의

HTTP(HyperText Transfer Protocol)는 WWW(World Wide Web)에서 HTML 문서를 주고받을 수 있는 프로토콜. 주로 TCP를 사용하고 HTTP/3부터는 UDP를 사용하며, 80번 포트를 사용한다. HTTP는 클라이언트와 서버 사이에 이루어지는 요청(Request)/응답(Response) 프로토콜. HTTP를 통해 전달되는 자료는 http://로 시작하는 URL(인터넷 주소)로 조회 가능.

2
HTTP 핵심 특징
★★★
특징설명한계 및 보완
비연결성
(Connectionless)		 클라이언트가 요청을 보내고 서버가 응답하면 연결을 끊는다. HTTP/1.0에서 기본 방식. 요청마다 새로운 연결이 필요. HTTP/1.1에서 Keep-Alive(지속 연결)로 보완. 하나의 TCP 연결로 여러 요청/응답 처리.
무상태성
(Stateless)		 서버가 클라이언트의 이전 상태를 기억하지 않는다. 각 요청은 독립적. 이전 요청과의 관계 없음. 전자상거래·장바구니처럼 일관된 방식으로 상호작용하길 원할 때 문제. Cookie, Session, JWT로 상태 유지.

HTTP 버전 비교 — 1.0 / 1.1 / 2.0
★★★ 2020 국가직 출제

HTTP 버전은 시험에서 RFC 번호·주요 기능·연결 방식 위주로 출제돼요. 특히 HTTP 1.0과 1.1의 RFC 번호, 1.1에서 추가된 지속 연결(Keep-Alive)과 파이프라이닝이 자주 나와요. 2020 국가직 문19에서 버전·RFC가 출제됐어요.

HTTP/0.9
(1991)
GET 메서드만 존재. 단순 HTML 전송만 가능.
헤더 없음. 상태 코드 없음.
현재 사용 안 함.
HTTP/1.0
(1996)
RFC: RFC 1945
연결: 요청마다 TCP 연결을 새로 생성하고 종료. 비연결성 기본.
특징: 헤더 추가. 상태 코드 추가. GET·POST·HEAD 메서드. Content-Type 지원.
단점: 매 요청마다 연결/해제 → 성능 저하. RTT(Round Trip Time) 증가.
HTTP/1.1
(1997)
RFC: RFC 2616
지속 연결(Keep-Alive): 하나의 TCP 연결로 여러 요청/응답 처리.
파이프라이닝(Pipelining): 응답을 기다리지 않고 요청을 여러 개 연속 전송.
Host 헤더 필수: 하나의 IP에 여러 도메인 운영 가능(가상 호스팅).
추가 메서드: PUT·PATCH·DELETE·OPTIONS·CONNECT 추가.
캐시 제어 강화: ETag, Cache-Control 헤더.
단점: HOL(Head-Of-Line) Blocking — 앞 요청이 막히면 뒤 요청도 지연.
HTTP/2
(2015)
RFC: RFC 7540
바이너리 프로토콜: 텍스트 대신 이진 데이터로 전송 → 파싱 효율 향상.
멀티플렉싱(Multiplexing): 하나의 연결로 여러 요청/응답을 동시 처리. HOL Blocking 해결.
헤더 압축(HPACK): 중복 헤더 압축 전송.
서버 푸시(Server Push): 클라이언트 요청 없이 서버가 리소스 미리 전송.
스트림: 하나의 TCP 연결에서 여러 스트림 동시 처리.

1
HTTP 요청 메서드 (Method)
★★★
메서드 정의

HTTP 메서드는 클라이언트가 서버에게 어떤 동작을 원하는지 알려주는 방식. HTTP/1.0에서는 GET·POST·HEAD만 존재했고, HTTP/1.1에서 PUT·DELETE·OPTIONS·CONNECT·PATCH 등이 추가됐다. 각 메서드의 특성(안전성·멱등성)을 이해해야 해요.

GET
서버의 리소스를 조회할 때 사용. 데이터를 URL에 포함하여 전송. 요청 바디 없음. 가장 기본적인 메서드. 북마크·캐싱 가능.
POST
서버에 데이터를 전송(생성)할 때 사용. 데이터를 요청 바디에 포함. 폼 데이터, 파일, JSON 등 다양한 데이터 전송 가능. URL에 데이터가 노출되지 않음.
PUT
서버의 리소스를 전체 수정(대체)할 때 사용. 리소스가 없으면 생성. 멱등성(Idempotent) — 여러 번 요청해도 결과 동일.
DELETE
서버의 리소스를 삭제할 때 사용. 멱등성 있음. 성공 시 200 OK 또는 204 No Content 반환.
PATCH
리소스의 일부만 수정할 때 사용. PUT은 전체 대체, PATCH는 부분 수정. HTTP/1.1에서 추가.
HEAD
GET과 동일하나 응답 바디 없이 헤더만 반환. 리소스 존재 확인, 크기 확인 등에 사용. HTTP/1.0부터 존재.
OPTIONS
서버가 지원하는 메서드 목록 확인. CORS(교차 출처 리소스 공유) 사전 요청에 사용. HTTP/1.1 추가.
CONNECT
프록시를 통해 터널 연결 설정. HTTPS 연결 시 프록시 서버에 터널 요청. HTTP/1.1 추가.

멱등성(Idempotency): 동일한 요청을 여러 번 반복해도 결과가 동일한 성질. GET·PUT·DELETE·HEAD·OPTIONS는 멱등. POST·PATCH는 비멱등 (호출할 때마다 새 결과 생성 또는 누적).

501 = Not Implemented (서버가 기능을 지원하지 않음). "서버 과부하로 서비스 불가" = 503 Service Unavailable. 숫자 하나 차이지만 의미가 완전히 달라요.

"401: 권한 없음 — 접속 실패, 로그온 하려는 요청에서 패스워드가 맞지 않을 경우 발생" → ✓ 맞음

401 = Unauthorized (인증 실패). 403 = Forbidden (인증은 됐지만 권한 없음). 두 코드의 차이도 자주 출제돼요.

HTTPS (HTTP Secure) 정의 및 원리
★★★
HTTPS 정의

HTTPS(HTTP Secure)는 SSL(Secure Socket Layer) 또는 TLS(Transport Layer Security)를 이용하여 클라이언트와 서버 사이에 주고받는 정보를 보호하는 데 사용되는 프로토콜. HTTP와 SSL/TLS를 합친 것으로 웹 브라우저와 웹 서버 사이에 안전한 통신을 구현한다. 기본 포트: 443.

HTTPS를 통해 전달되는 자료는 URL이 https://로 시작해요. HTTPS는 HTTP에 암호화·인증·무결성을 추가한 것으로, 도청·중간자 공격·위장에 대응해요. 브라우저 주소창의 🔒 자물쇠 아이콘이 HTTPS를 나타내요.

HTTPS 통신 흐름: ① 클라이언트가 서버에 접속 → ② SSL/TLS 핸드셰이크(인증서 확인·키 교환) → ③ 대칭키로 암호화된 통신 → ④ 연결 종료. 핸드셰이크에서 서버 인증서(X.509)를 통해 서버가 진짜임을 확인해요.

HTTP와 보안 계층 위치
★★
HTTP vs HTTPS 프로토콜 스택 비교
시험 빈출 포인트: HTTPS = HTTP + SSL/TLS. 포트 = 443. HTTPS 사용 시 URL은 https://로 시작. HTTPS, SMTPS, FTPS 모두 TLS를 지원. SSL은 TLS 이전 버전이며 현재는 TLS가 표준.

기출문제 전수 해설 — 3문제
연도순
📄 2020 국가직 9급 정보시스템보안 문19 · HTTP 버전·RFC·코드★★★
HTTP에 대한 설명으로 옳지 않은 것은?
① 상태 코드 404는 클라이언트의 PUT 요청이 성공적이라는 것을 의미한다.
② HTTP 1.0 프로토콜은 RFC 1945이고, HTTP 1.1 프로토콜은 RFC 2616에 기술되어 있다.
③ HTTPS는 SSL을 이용하여 클라이언트와 서버 사이에 주고받는 정보를 보호하는 데 사용된다.
④ Request는 웹서버에 데이터를 요청하거나 전송할 때 보내는 패킷으로 GET, POST와 같은 메소드를 사용한다.
정답 ① — 404 = Not Found (요청한 URL을 찾을 수 없음). PUT 성공 = 200 OK 또는 201 Created. ②RFC 1945(1.0)·RFC 2616(1.1) 모두 맞음. ③HTTPS = HTTP + SSL/TLS 맞음. ④GET·POST 메서드 설명 맞음.

② 404 — 서버에 요구된 URL을 찾을 수 없음(Not Found)
③ 500 — 서버 내부오류가 발생하여 더는 진행할 수 없음(Internal Server Error)
④ 501 — 서버 과부하로 서비스할 수 없음(Service Unavailable)
정답 ④ — 501 = Not Implemented (서버가 요청 기능을 지원하지 않음). "서버 과부하로 서비스할 수 없음(Service Unavailable)" = 503. 501과 503은 숫자가 비슷해 혼동하기 쉬운 대표적인 함정.

HTTP 프로토콜에서 요청 메시지에 적합한 인증 부족(Unauthorized)이라는 클라이언트 오류를 의미하는 상태 코드는?
① 200
② 301
③ 401
④ 503
정답 ③ 401 — 401 = Unauthorized (인증 부족). 200=OK(성공). 301=Moved Permanently(영구 이동). 503=Service Unavailable(서비스 불가). 클라이언트 오류(4xx) 중 인증 관련 = 401.
